📨 ارتباط Asynchronous

Event چیست؟

📅 به‌روزرسانی: 1405/04/26
⏱️ زمان مطالعه: ۶ دقیقه
👤 تیم فنی

در معماری Microservices، Event پیامی است که اعلام می‌کند اتفاقی در گذشته رخ داده است.

برای مثال، زمانی که یک سفارش ثبت می‌شود، سرویس Order می‌تواند رویدادی با نام OrderCreated منتشر کند.

🎯
Event به سرویس‌های دیگر نمی‌گوید چه کاری انجام دهند؛ فقط آن‌ها را از وقوع یک اتفاق مطلع می‌کند.

یک مثال ساده

فرض کنید کاربر یک سفارش جدید ثبت می‌کند.

پس از ثبت موفق سفارش، سرویس Order یک Event منتشر می‌کند و سایر سرویس‌ها بر اساس نیاز خود به آن واکنش نشان می‌دهند.

OrderCreated Event 

Order Service

      │

OrderCreated

      ▼

Message Broker

      │

      ├──► Email Service

      ├──► Inventory Service

      └──► Analytics Service

Email Service ایمیل تأیید ارسال می‌کند، Inventory Service موجودی را به‌روزرسانی می‌کند و Analytics Service اطلاعات فروش را ثبت می‌کند.

✅
سرویس Order لازم نیست Consumerها را بشناسد و فقط وقوع Event را اعلام می‌کند.

ساختار یک Event

یک Event معمولاً شامل نام رویداد، شناسه یکتا، زمان وقوع و اطلاعات موردنیاز Consumerها است.

Event Example 

{
  "eventId": "55d9c681",
  "eventType": "OrderCreated",
  "occurredAt": "2026-07-16T11:20:00Z",
  "orderId": 1001,
  "customerId": 25,
  "totalAmount": 850000
}
💡
نام Event بهتر است به شکل گذشته نوشته شود؛ مانند OrderCreated، PaymentCompleted یا ProductPriceChanged.

تفاوت Event و Command

Event و Command هر دو نوعی Message هستند، اما هدف متفاوتی دارند.

ویژگی Event Command
هدف اعلام وقوع یک اتفاق درخواست انجام یک عملیات
زمان فعل گذشته دستوری
نمونه نام OrderCreated CreateOrder
دریافت‌کننده ممکن است چند Consumer باشد معمولاً یک Handler مشخص
انتظار نتیجه معمولاً ندارد ممکن است نتیجه مورد انتظار باشد
⚠️
Event نباید به شکل دستور طراحی شود. نامی مانند SendOrderEmail یک Command است، اما OrderCreated یک Event محسوب می‌شود.

Event چه تفاوتی با Message دارد؟

Message یک اصطلاح عمومی برای داده‌ای است که میان سرویس‌ها منتقل می‌شود.

Event یکی از انواع Message است که بیان می‌کند یک اتفاق قبلاً رخ داده است.

Message Types 

Message

   │

   ├── Command
   │     └── CreatePayment
   │
   └── Event
         └── PaymentCompleted

Integration Event

Eventهایی که میان Microserviceها منتشر می‌شوند معمولاً با نام Integration Event شناخته می‌شوند.

این Eventها بخشی از قرارداد عمومی سرویس هستند و Consumerهای دیگر به ساختار آن‌ها وابسته می‌شوند.

Integration Event 

Payment Service

      │

PaymentCompleted

      ▼

Message Broker

      │

      ├──► Order Service

      ├──► Notification Service

      └──► Accounting Service
📌
تغییر ساختار Integration Event باید با دقت انجام شود؛ زیرا ممکن است چند سرویس دیگر از آن استفاده کنند.

مزایای استفاده از Event

  • کاهش وابستگی مستقیم میان سرویس‌ها
  • امکان اضافه شدن Consumer جدید بدون تغییر Producer
  • اجرای پردازش‌ها به‌صورت Asynchronous
  • افزایش توسعه‌پذیری سیستم
  • مناسب برای ثبت تغییرات مهم Business

چالش‌های Event

چالش توضیح
Duplicate Event ممکن است یک Event بیش از یک‌بار تحویل داده شود.
Event Ordering ترتیب رسیدن Eventها همیشه تضمین‌شده نیست.
Eventual Consistency سرویس‌ها ممکن است با کمی تأخیر هماهنگ شوند.
Schema Change تغییر قرارداد Event ممکن است Consumerها را خراب کند.
Tracing ردیابی جریان کامل عملیات دشوارتر از ارتباط Sync است.
⚠️
Consumer باید در برابر دریافت Event تکراری مقاوم باشد و از Idempotency استفاده کند.

Best Practiceهای طراحی Event

انجام دهید ✅ انجام ندهید ❌
نام Event را در زمان گذشته انتخاب کنید. از نام‌های دستوری استفاده نکنید.
شناسه و زمان وقوع Event را قرار دهید. به Timestamp سیستم Consumer وابسته نشوید.
فقط اطلاعات موردنیاز را ارسال کنید. Entity کامل Database را منتشر نکنید.
قرارداد Event را Version کنید. ساختار Event را ناگهانی تغییر ندهید.
Consumer را Idempotent طراحی کنید. فرض نکنید Event فقط یک‌بار تحویل می‌شود.

اشتباه رایج

❌
یک اشتباه رایج این است که Producer بداند هر Consumer دقیقاً چه کاری انجام می‌دهد. Producer فقط باید Event را منتشر کند و تصمیم‌گیری درباره واکنش به آن بر عهده Consumerها باشد.

نکته مصاحبه

🎓
سؤال: تفاوت Event و Command چیست؟

پاسخ: Command درخواست انجام یک عملیات است و معمولاً یک Handler مشخص دارد؛ اما Event اعلام می‌کند اتفاقی رخ داده و ممکن است توسط چند Consumer مستقل پردازش شود.

واژه‌نامه

اصطلاح توضیح
Event پیامی که وقوع یک اتفاق در گذشته را اعلام می‌کند
Command پیامی برای درخواست انجام یک عملیات
Integration Event Event منتشرشده میان Microserviceها
Producer سرویسی که Event را منتشر می‌کند
Consumer سرویسی که Event را دریافت و پردازش می‌کند
Eventual Consistency هماهنگ شدن داده‌های سرویس‌ها با کمی تأخیر

قانون طلایی

🎯
Event باید بیان کند چه اتفاقی رخ داده است، نه اینکه سرویس‌های دیگر باید چه کاری انجام دهند.

جمع‌بندی

Event پیامی است که وقوع یک اتفاق را اعلام می‌کند. Producer آن را منتشر می‌کند و Consumerهای مختلف می‌توانند بدون ایجاد وابستگی مستقیم به آن واکنش نشان دهند.

استفاده صحیح از Event باعث افزایش انعطاف‌پذیری و کاهش Coupling می‌شود؛ اما مدیریت Eventهای تکراری، ترتیب پردازش، تغییر قرارداد و Eventual Consistency اهمیت زیادی دارد.

➡️
در بخش بعد با Event-Driven Architecture آشنا می‌شویم و بررسی می‌کنیم چگونه سرویس‌ها با انتشار و دریافت Event با یکدیگر همکاری می‌کنند.

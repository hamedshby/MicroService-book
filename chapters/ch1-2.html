1.2 Communication بین Microserviceها

📅 به‌روزرسانی: ۱۴۰۵/۰۴/۱۶
⏱️ زمان مطالعه: ۱۵ دقیقه
👤 تیم فنی

یکی از مهم‌ترین تفاوت‌های معماری Monolith و Microservices، نحوه ارتباط بین بخش‌های مختلف سیستم است. در معماری Monolith، تمام ماژول‌ها در یک Process اجرا می‌شوند و برای برقراری ارتباط تنها کافی است یک متد فراخوانی شود.

Monolith 
OrderService.ProcessOrder()

        │

        ▼

PaymentService.Pay()

        │

        ▼

InventoryService.Reserve()

اما در معماری Microservices هر سرویس در یک Process مستقل، و معمولاً روی یک ماشین یا Container جداگانه اجرا می‌شود. در نتیجه دیگر امکان فراخوانی مستقیم متدها وجود ندارد و سرویس‌ها باید از طریق شبکه با یکدیگر ارتباط برقرار کنند.

⚠️
در Microservices هر ارتباط، یک ارتباط شبکه‌ای (Network Call) است و هر ارتباط شبکه‌ای می‌تواند با خطا، Timeout، قطع ارتباط یا کندی شبکه مواجه شود.

Communication چیست؟

Communication به مجموعه روش‌هایی گفته می‌شود که سرویس‌های مختلف برای تبادل اطلاعات، درخواست عملیات یا اطلاع‌رسانی رویدادها از آن استفاده می‌کنند.

به عنوان مثال در یک فروشگاه اینترنتی، هنگام ثبت سفارش ممکن است سرویس سفارش نیاز داشته باشد:

  • موجودی کالا را بررسی کند.
  • پرداخت را انجام دهد.
  • پیامک ارسال کند.
  • ایمیل تأیید سفارش ارسال کند.
  • امتیاز باشگاه مشتریان را افزایش دهد.

تمام این عملیات نیازمند برقراری ارتباط بین چندین Microservice هستند.

Synchronous Communication

در ارتباط Synchronous، سرویس ارسال‌کننده درخواست (Client) تا زمانی که پاسخ از سرویس مقصد دریافت نشود، منتظر می‌ماند. به عبارت دیگر، فرآیند اجرای برنامه تا دریافت نتیجه متوقف خواهد شد.

این مدل دقیقاً مشابه فراخوانی یک متد در برنامه‌های Monolith است؛ با این تفاوت که این بار فراخوانی از طریق شبکه انجام می‌شود و ممکن است با مشکلاتی مانند Latency، Timeout یا قطع ارتباط مواجه شود.

روش‌های پیاده‌سازی ارتباط Synchronous

در معماری Microservices معمولاً از سه روش اصلی برای ارتباط همزمان استفاده می‌شود.

روش پروتکل کاربرد
REST API HTTP / HTTPS عمومی‌ترین روش ارتباط
gRPC HTTP/2 ارتباط سریع بین سرویس‌ها
GraphQL HTTP انعطاف‌پذیری در دریافت داده
✅
در پروژه‌های ASP.NET Core معمولاً REST و gRPC پرکاربردترین روش‌های ارتباط Synchronous هستند.

مزایای ارتباط Synchronous

  • پیاده‌سازی ساده و قابل فهم
  • دریافت پاسخ در همان لحظه
  • مناسب برای عملیات‌های کوتاه
  • دیباگ کردن آسان‌تر
  • مدیریت خطا ساده‌تر
  • عدم نیاز به Message Broker
ℹ️
اگر نتیجه عملیات بلافاصله مورد نیاز باشد، ارتباط Synchronous بهترین انتخاب است.

معایب ارتباط Synchronous

با وجود سادگی، این نوع ارتباط مشکلات مهمی نیز دارد.

۱- وابستگی زمانی (Temporal Coupling)

هر دو سرویس باید همزمان در دسترس باشند. اگر سرویس مقصد Down باشد، سرویس ارسال‌کننده نیز قادر به ادامه پردازش نخواهد بود.

⚠️
یکی از مهم‌ترین دلایل استفاده از Message Brokerها، کاهش همین وابستگی زمانی بین سرویس‌ها است.

مثال واقعی

فرض کنید کاربر دکمه ثبت سفارش را فشار می‌دهد.

سرویس Order برای ثبت سفارش باید مراحل زیر را انجام دهد.

  1. بررسی موجودی کالا
  2. دریافت مبلغ از درگاه پرداخت
  3. ثبت سفارش
  4. ارسال پاسخ به کاربر

در این سناریو کاربر باید نتیجه پرداخت را همان لحظه مشاهده کند؛ بنابراین ارتباط همزمان (Synchronous) بهترین انتخاب خواهد بود.

🎯
هر زمان کاربر نیاز داشته باشد نتیجه عملیات را بلافاصله مشاهده کند، معمولاً از ارتباط‌های Synchronous استفاده می‌شود.

Asynchronous Communication

در ارتباط Asynchronous، سرویس ارسال‌کننده پس از ارسال پیام، منتظر پاسخ نمی‌ماند و اجرای برنامه را ادامه می‌دهد. سرویس مقصد نیز هر زمان که آماده باشد پیام را دریافت و پردازش می‌کند.

برخلاف ارتباط‌های Synchronous، دو سرویس لازم نیست هم‌زمان در دسترس باشند. پیام ابتدا در یک Message Broker ذخیره می‌شود و سپس توسط Consumer دریافت خواهد شد.

چرا Asynchronous Communication؟

با بزرگ‌تر شدن سیستم، استفاده از ارتباط‌های Sync برای تمام عملیات‌ها باعث کاهش Performance و افزایش وابستگی بین سرویس‌ها می‌شود.

فرض کنید پس از ثبت سفارش، عملیات‌های زیر باید انجام شوند.

  • ارسال ایمیل
  • ارسال پیامک
  • به‌روزرسانی انبار
  • ثبت لاگ
  • ثبت اطلاعات در سیستم گزارش‌گیری
  • ارسال Notification
  • محاسبه امتیاز باشگاه مشتریان

اگر تمام این عملیات‌ها به صورت Sync انجام شوند، کاربر باید تا پایان اجرای همه آن‌ها منتظر بماند؛ در حالی که بیشتر این عملیات‌ها برای نمایش نتیجه به کاربر ضروری نیستند.

هرچه تعداد سرویس‌های Sync بیشتر شود، احتمال Timeout، کاهش Performance و Cascading Failure نیز بیشتر خواهد شد.

استفاده از Message Broker

در معماری‌های مدرن، معمولاً سرویس Order تنها پیام "Order Created" را در Message Broker منتشر می‌کند.

سپس سایر سرویس‌ها بدون وابستگی مستقیم، پیام را دریافت و عملیات مربوط به خود را انجام می‌دهند.

مزایای ارتباط Asynchronous

  • کاهش وابستگی بین سرویس‌ها
  • افزایش سرعت پاسخ به کاربر
  • مقیاس‌پذیری بهتر
  • امکان پردازش موازی پیام‌ها
  • افزایش تحمل خطا (Fault Tolerance)
  • امکان Retry خودکار
  • امکان استفاده از Dead Letter Queue
  • امکان پردازش در زمان‌های مختلف

معایب ارتباط Asynchronous

  • پیچیدگی بیشتر نسبت به REST
  • اشکال‌زدایی دشوارتر
  • عدم دریافت پاسخ فوری
  • نیاز به Message Broker
  • مدیریت Duplicate Message
  • مدیریت Ordering پیام‌ها
  • نیاز به الگوهایی مانند Outbox و Inbox
⚠️
استفاده از Message Broker تمام مشکلات را حل نمی‌کند؛ بلکه نوع مشکلات را تغییر می‌دهد. در سیستم‌های Event Driven باید موضوعاتی مانند Duplicate، Ordering، Retry، Idempotency و Eventually Consistent نیز مدیریت شوند.

Event Driven Architecture

در معماری Event Driven، ارتباط بین سرویس‌ها بر اساس وقوع یک رویداد (Event) انجام می‌شود.

به جای اینکه یک سرویس مستقیماً سرویس دیگری را فراخوانی کند، تنها وقوع یک رویداد را اعلام می‌کند.

در این مدل، سرویس Order هیچ اطلاعی از سرویس‌های دریافت‌کننده ندارد. تنها یک Event منتشر می‌کند و هر سرویسی که به آن علاقه‌مند باشد، می‌تواند آن را مصرف کند.

📌
اصل مهم در معماری Event Driven این است که Publisher نباید Consumerهای خود را بشناسد.

Message Driven Architecture

در معماری Message Driven، تمرکز روی تبادل پیام بین سرویس‌ها است. این پیام ممکن است یک Command، یک Event یا حتی یک Response باشد.

برخلاف Event Driven که مفهوم اصلی آن «اتفاق افتادن یک رویداد» است، در Message Driven هر نوع پیامی می‌تواند بین سرویس‌ها رد و بدل شود.

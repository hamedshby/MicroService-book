انواع Exchange در RabbitMQ

📅 به‌روزرسانی: 1405/04/26
⏱️ زمان مطالعه: ۱۲ دقیقه
👤 تیم فنی

در RabbitMQ، سرویس Producer پیام را مستقیماً برای Queue ارسال نمی‌کند. پیام ابتدا وارد Exchange می‌شود و Exchange آن را بر اساس قوانین Routing به یک یا چند Queue هدایت می‌کند.

RabbitMQ چهار نوع اصلی Exchange دارد:

  • Direct Exchange
  • Fanout Exchange
  • Topic Exchange
  • Headers Exchange
🎯
در تمام مثال‌های این فصل، Order Service پیام‌های مربوط به سفارش را منتشر می‌کند و سرویس‌هایی مانند Notification، Payment و Analytics آن‌ها را دریافت می‌کنند.

نصب پکیج RabbitMQ در .NET

ابتدا پکیج رسمی RabbitMQ را به پروژه Order Service اضافه کنید.

Terminal 
dotnet add package RabbitMQ.Client

سپس Connection و Channel را ایجاد می‌کنیم. در مثال‌های بعدی از همین ساختار استفاده خواهد شد.

C# - Connection 
using RabbitMQ.Client;

var factory = new ConnectionFactory
{
    HostName = "localhost",
    UserName = "guest",
    Password = "guest",
    ClientProvidedName = "OrderService"
};

await using var connection = await factory.CreateConnectionAsync();
await using var channel = await connection.CreateChannelAsync();
⚠️
در برنامه واقعی نباید برای انتشار هر پیام یک Connection جدید ایجاد کنید. Connection باید طول عمر بیشتری داشته باشد و مدیریت آن از طریق Dependency Injection انجام شود.

Direct Exchange

در Direct Exchange پیام زمانی وارد Queue می‌شود که Routing Key پیام دقیقاً با Binding Key آن Queue برابر باشد.

Direct Exchange Flow 

Order Service

Routing Key: order.created

        │

        ▼

Direct Exchange

        │

        ├── order.created
        │       ▼
        │   Notification Queue
        │
        └── order.cancelled
                ▼
            Refund Queue

تعریف Direct Exchange و Queue

C# - Direct Exchange Setup 
const string exchangeName = "order.direct";
const string queueName = "order.notification.queue";
const string routingKey = "order.created";

await channel.ExchangeDeclareAsync(
    exchange: exchangeName,
    type: ExchangeType.Direct,
    durable: true,
    autoDelete: false);

await channel.QueueDeclareAsync(
    queue: queueName,
    durable: true,
    exclusive: false,
    autoDelete: false);

await channel.QueueBindAsync(
    queue: queueName,
    exchange: exchangeName,
    routingKey: routingKey);

انتشار پیام OrderCreated

C# - Direct Exchange Publisher 
using System.Text;
using System.Text.Json;

var message = new
{
    MessageId = Guid.NewGuid(),
    MessageType = "OrderCreated",
    OrderId = 1001,
    CustomerId = 25,
    TotalAmount = 850000,
    CreatedAt = DateTime.UtcNow
};

byte[] body = Encoding.UTF8.GetBytes(
    JsonSerializer.Serialize(message));

await channel.BasicPublishAsync(
    exchange: exchangeName,
    routingKey: "order.created",
    body: body);
📌
Direct Exchange برای سناریوهایی مناسب است که مقصد پیام با یک Routing Key دقیق مانند order.created یا order.cancelled مشخص می‌شود.

Fanout Exchange

Fanout Exchange پیام را برای تمام Queueهای متصل به خود ارسال می‌کند و Routing Key را نادیده می‌گیرد.

Fanout Exchange Flow 

Order Service

      │

OrderCreated

      ▼

Fanout Exchange

      │

      ├──► Notification Queue

      ├──► Analytics Queue

      └──► Audit Queue

تعریف Fanout Exchange

C# - Fanout Exchange Setup 
const string exchangeName = "order.fanout";

await channel.ExchangeDeclareAsync(
    exchange: exchangeName,
    type: ExchangeType.Fanout,
    durable: true,
    autoDelete: false);

string[] queues =
[
    "order.notification.queue",
    "order.analytics.queue",
    "order.audit.queue"
];

foreach (string queueName in queues)
{
    await channel.QueueDeclareAsync(
        queue: queueName,
        durable: true,
        exclusive: false,
        autoDelete: false);

    await channel.QueueBindAsync(
        queue: queueName,
        exchange: exchangeName,
        routingKey: string.Empty);
}

انتشار پیام

C# - Fanout Exchange Publisher 
byte[] body = Encoding.UTF8.GetBytes(
    JsonSerializer.Serialize(message));

await channel.BasicPublishAsync(
    exchange: exchangeName,
    routingKey: string.Empty,
    body: body);
✅
در این مثال، Notification، Analytics و Audit هرکدام یک نسخه مستقل از پیام OrderCreated را دریافت می‌کنند.

Topic Exchange

Topic Exchange پیام‌ها را با استفاده از الگوهای Routing Key مسیریابی می‌کند.

Routing Key معمولاً از چند بخش تشکیل می‌شود که با نقطه از یکدیگر جدا شده‌اند.

Routing Keys 

order.created

order.paid

order.cancelled

order.payment.failed
علامت کاربرد
* دقیقاً یک بخش را پوشش می‌دهد.
# صفر یا چند بخش را پوشش می‌دهد.

تعریف Topic Exchange

C# - Topic Exchange Setup 
const string exchangeName = "order.topic";

await channel.ExchangeDeclareAsync(
    exchange: exchangeName,
    type: ExchangeType.Topic,
    durable: true,
    autoDelete: false);

await channel.QueueDeclareAsync(
    queue: "order.payment.queue",
    durable: true,
    exclusive: false,
    autoDelete: false);

await channel.QueueBindAsync(
    queue: "order.payment.queue",
    exchange: exchangeName,
    routingKey: "order.payment.*");

await channel.QueueDeclareAsync(
    queue: "order.audit.queue",
    durable: true,
    exclusive: false,
    autoDelete: false);

await channel.QueueBindAsync(
    queue: "order.audit.queue",
    exchange: exchangeName,
    routingKey: "order.#");

انتشار خطای پرداخت سفارش

C# - Topic Exchange Publisher 
var paymentFailedMessage = new
{
    MessageId = Guid.NewGuid(),
    MessageType = "OrderPaymentFailed",
    OrderId = 1001,
    Reason = "InsufficientBalance",
    CreatedAt = DateTime.UtcNow
};

byte[] body = Encoding.UTF8.GetBytes(
    JsonSerializer.Serialize(paymentFailedMessage));

await channel.BasicPublishAsync(
    exchange: exchangeName,
    routingKey: "order.payment.failed",
    body: body);

این پیام به Queueهایی که دارای Binding Key برابر order.payment.* یا order.# هستند ارسال می‌شود.

⚠️
الگوی order.* فقط پیام‌هایی مانند order.created را پوشش می‌دهد و با order.payment.failed مطابقت ندارد؛ زیرا پیام دارای سه بخش است.

Headers Exchange

در Headers Exchange مسیریابی بر اساس Headerهای پیام انجام می‌شود و Routing Key نقشی ندارد.

برای مثال می‌توان سفارش‌ها را بر اساس کشور یا اولویت آن‌ها مسیریابی کرد.

Headers Exchange Flow 

Order Service

Headers:
Country = IR
Priority = High

      │

      ▼

Headers Exchange

      │

      ├──► Iran Orders Queue

      └──► High Priority Queue

تعریف Headers Exchange

C# - Headers Exchange Setup 
const string exchangeName = "order.headers";
const string queueName = "order.high-priority.queue";

await channel.ExchangeDeclareAsync(
    exchange: exchangeName,
    type: ExchangeType.Headers,
    durable: true,
    autoDelete: false);

await channel.QueueDeclareAsync(
    queue: queueName,
    durable: true,
    exclusive: false,
    autoDelete: false);

var bindingArguments =
    new Dictionary<string, object?>
    {
        ["x-match"] = "all",
        ["country"] = "IR",
        ["priority"] = "high"
    };

await channel.QueueBindAsync(
    queue: queueName,
    exchange: exchangeName,
    routingKey: string.Empty,
    arguments: bindingArguments);

انتشار پیام دارای Header

C# - Headers Exchange Publisher 
var properties = new BasicProperties
{
    Persistent = true,
    ContentType = "application/json",
    Headers = new Dictionary<string, object?>
    {
        ["country"] = "IR",
        ["priority"] = "high"
    }
};

byte[] body = Encoding.UTF8.GetBytes(
    JsonSerializer.Serialize(message));

await channel.BasicPublishAsync(
    exchange: exchangeName,
    routingKey: string.Empty,
    mandatory: false,
    basicProperties: properties,
    body: body);
💡
Headers Exchange نسبت به Direct، Fanout و Topic کمتر استفاده می‌شود و زمانی مناسب است که Routing به چند ویژگی پیام وابسته باشد.

Default Exchange

RabbitMQ یک Direct Exchange داخلی با نام خالی "" دارد که با نام Default Exchange شناخته می‌شود.

در این حالت، نام Queue به‌عنوان Routing Key ارسال می‌شود.

C# - Default Exchange 
const string queueName = "order.notification.queue";

await channel.QueueDeclareAsync(
    queue: queueName,
    durable: true,
    exclusive: false,
    autoDelete: false);

byte[] body = Encoding.UTF8.GetBytes(
    JsonSerializer.Serialize(message));

await channel.BasicPublishAsync(
    exchange: string.Empty,
    routingKey: queueName,
    body: body);
⚠️
Default Exchange برای مثال‌های ساده مناسب است؛ اما در سیستم‌های بزرگ بهتر است Exchange و قوانین Routing به‌صورت واضح تعریف شوند.

نمونه Consumer در .NET

Consumer پیام OrderCreated را از Notification Queue دریافت کرده و پس از پردازش موفق، پیام را Ack می‌کند.

C# - Notification Consumer 
using RabbitMQ.Client.Events;
using System.Text;

const string queueName = "order.notification.queue";

var consumer = new AsyncEventingBasicConsumer(channel);

consumer.ReceivedAsync += async (_, eventArgs) =>
{
    try
    {
        string json = Encoding.UTF8.GetString(
            eventArgs.Body.Span);

        Console.WriteLine(
            $"OrderCreated received: {json}");

        // Send order notification...

        await channel.BasicAckAsync(
            deliveryTag: eventArgs.DeliveryTag,
            multiple: false);
    }
    catch (Exception exception)
    {
        Console.WriteLine(exception.Message);

        await channel.BasicNackAsync(
            deliveryTag: eventArgs.DeliveryTag,
            multiple: false,
            requeue: false);
    }
};

await channel.BasicConsumeAsync(
    queue: queueName,
    autoAck: false,
    consumer: consumer);
✅
با قرار دادن autoAck روی false، پیام فقط پس از اجرای BasicAckAsync به‌عنوان پردازش‌شده تأیید می‌شود.

مقایسه انواع Exchange

Exchange روش Routing مثال Order Service
Direct تطابق دقیق Routing Key ارسال order.cancelled به Refund Queue
Fanout ارسال به تمام Queueها انتشار OrderCreated برای چند سرویس
Topic تطابق الگویی Routing Key مدیریت پیام‌های order.payment.*
Headers Headerهای پیام مسیریابی سفارش بر اساس کشور و اولویت

کدام Exchange را انتخاب کنیم؟

  • برای مسیریابی دقیق از Direct Exchange استفاده کنید.
  • برای ارسال یک Event به تمام Subscriberها از Fanout Exchange استفاده کنید.
  • برای مسیریابی انعطاف‌پذیر و الگومحور از Topic Exchange استفاده کنید.
  • برای مسیریابی بر اساس چند ویژگی پیام از Headers Exchange استفاده کنید.

اشتباه رایج

❌
تنها تعریف Exchange کافی نیست. باید Queue و Binding نیز قبل از انتشار پیام ایجاد شده باشند؛ در غیر این صورت ممکن است پیام هیچ Queue مقصدی نداشته باشد.

نکته مصاحبه

🎓
سؤال: تفاوت Direct و Topic Exchange چیست؟

پاسخ: Direct Exchange به تطابق دقیق Routing Key و Binding Key نیاز دارد؛ اما Topic Exchange امکان مسیریابی الگومحور با استفاده از * و # را فراهم می‌کند.

قانون طلایی

🎯
Exchange را بر اساس نیاز واقعی Routing انتخاب کنید. پیچیده‌تر بودن Exchange به معنی بهتر بودن آن نیست.

جمع‌بندی

Exchange مسئول مسیریابی پیام‌ها در RabbitMQ است. Direct از تطابق دقیق، Fanout از Broadcast، Topic از الگوهای Routing و Headers از Headerهای پیام استفاده می‌کند.

در .NET می‌توان با استفاده از RabbitMQ.Client، Exchange و Queue را تعریف کرد، Binding ساخت، پیام منتشر کرد و آن را با AsyncEventingBasicConsumer دریافت کرد.

➡️
در بخش بعد Queue را دقیق‌تر بررسی می‌کنیم و ویژگی‌هایی مانند Durable، Exclusive، Auto Delete و نگهداری پیام‌های OrderCreated را همراه با کد .NET یاد می‌گیریم.

Message Broker چیست؟

📅 به‌روزرسانی: 1405/04/25
⏱️ زمان مطالعه: ۸ دقیقه
👤 تیم فنی

در ارتباط‌های Synchronous، یک سرویس مستقیماً سرویس دیگری را فراخوانی کرده و تا دریافت پاسخ منتظر می‌ماند.

اما در ارتباط‌های Asynchronous، سرویس فرستنده پیام را ارسال می‌کند و برای ادامه پردازش، منتظر اجرای سرویس مقصد نمی‌ماند.

ابزاری که دریافت، نگهداری و انتقال این پیام‌ها را مدیریت می‌کند، Message Broker نام دارد.

🎯
Message Broker واسطه‌ای میان سرویس تولیدکننده پیام و سرویس مصرف‌کننده پیام است و وابستگی مستقیم میان آن‌ها را کاهش می‌دهد.

یک مثال ساده

فرض کنید کاربر سفارشی ثبت می‌کند. پس از ثبت سفارش باید ایمیل تأیید، پیامک و Notification نیز ارسال شوند.

در روش Sync، سرویس Order باید تمام سرویس‌ها را یکی پس از دیگری فراخوانی کند.

Synchronous Communication 

Order Service

      │

      ├── REST ──► Email Service

      ├── REST ──► SMS Service

      └── REST ──► Notification Service

در این حالت، کندی یا خرابی یکی از سرویس‌ها می‌تواند زمان ثبت سفارش را افزایش دهد یا کل عملیات را با خطا مواجه کند.

با استفاده از Message Broker، سرویس Order فقط یک پیام منتشر می‌کند.

Asynchronous Communication 

Order Service

      │

OrderCreated Message

      ▼

Message Broker

      │

      ├──► Email Service

      ├──► SMS Service

      └──► Notification Service
✅
پس از ثبت پیام در Broker، سرویس Order می‌تواند پاسخ را سریع‌تر به کاربر برگرداند و سایر پردازش‌ها در پس‌زمینه انجام شوند.

اجزای اصلی Message Broker

جزء وظیفه
Producer پیام را تولید و برای Broker ارسال می‌کند.
Message Broker پیام را دریافت، نگهداری و مسیریابی می‌کند.
Queue یا Topic محل قرارگیری پیام‌ها تا زمان پردازش است.
Consumer پیام را دریافت و پردازش می‌کند.

Message چیست؟

Message بسته‌ای از اطلاعات است که یک سرویس برای سرویس دیگری ارسال می‌کند.

برای مثال، پیام ثبت سفارش می‌تواند شامل شناسه سفارش، شناسه مشتری، مبلغ و زمان ثبت سفارش باشد.

OrderCreated Message 

{
  "eventId": "7c14a9df",
  "eventType": "OrderCreated",
  "orderId": 1001,
  "customerId": 25,
  "totalAmount": 850000,
  "createdAt": "2026-07-16T10:30:00Z"
}
💡
پیام بهتر است فقط اطلاعات موردنیاز Consumerها را داشته باشد و به ساختار داخلی Database سرویس تولیدکننده وابسته نباشد.

مزایای Message Broker

  • کاهش وابستگی مستقیم میان Microserviceها
  • انجام پردازش‌های زمان‌بر در پس‌زمینه
  • افزایش تحمل خطا و پایداری سیستم
  • امکان Retry پیام‌های ناموفق
  • امکان پردازش پیام‌ها توسط چند Consumer
  • کنترل بهتر حجم بالای درخواست‌ها

برای مثال، اگر Email Service موقتاً از دسترس خارج شود، پیام می‌تواند در Queue باقی بماند و پس از بازیابی سرویس پردازش شود.

چالش‌های Message Broker

استفاده از Message Broker فقط مزیت ندارد و پیچیدگی‌های جدیدی نیز وارد سیستم می‌کند.

چالش توضیح
Duplicate Message ممکن است یک پیام بیشتر از یک‌بار تحویل داده شود.
Message Ordering همیشه تضمینی برای ترتیب پردازش پیام‌ها وجود ندارد.
Eventual Consistency اطلاعات سرویس‌ها ممکن است با کمی تأخیر هماهنگ شوند.
Monitoring ردیابی جریان پیام‌ها نسبت به REST دشوارتر است.
Broker Failure خود Message Broker نیز باید پایدار و قابل بازیابی باشد.
⚠️
Consumerها باید به‌صورت Idempotent طراحی شوند؛ یعنی پردازش چندباره یک پیام نتیجه نهایی را خراب نکند.

Message Brokerهای معروف

ابزار کاربرد رایج
RabbitMQ Message Queue، Routing و پردازش وظایف
Apache Kafka Event Streaming و پردازش حجم بالای Event
Azure Service Bus پیام‌رسانی مدیریت‌شده در محیط Azure
Amazon SQS صف پیام مدیریت‌شده در AWS
📌
RabbitMQ و Kafka هر دو ابزار پیام‌رسانی هستند، اما معماری، نحوه نگهداری پیام و سناریوهای استفاده آن‌ها یکسان نیست.

چه زمانی از Message Broker استفاده کنیم؟

  • ارسال ایمیل، پیامک و Notification
  • پردازش فایل یا تصویر
  • ثبت گزارش و Audit Log
  • هماهنگی چند Microservice پس از یک رویداد
  • پردازش‌های زمان‌بر
  • مدیریت حجم ناگهانی درخواست‌ها

چه زمانی استفاده از آن ضروری نیست؟

  • عملیات بسیار ساده و کوتاه است.
  • پاسخ فوری سرویس مقصد موردنیاز است.
  • سیستم فقط از چند سرویس کوچک تشکیل شده است.
  • اضافه شدن Broker پیچیدگی بیشتری از ارزش آن ایجاد می‌کند.

اشتباه رایج

❌
ارسال Entity کامل Database داخل Message یک اشتباه رایج است. پیام باید یک قرارداد مستقل و مشخص داشته باشد تا تغییرات داخلی سرویس Producer باعث خرابی Consumerها نشود.

نکته مصاحبه

🎓
سؤال: اگر Consumer هنگام پردازش پیام از دسترس خارج شود چه اتفاقی می‌افتد؟

پاسخ: در یک طراحی صحیح، پیام تا زمانی که Consumer پردازش موفق آن را تأیید نکند از Queue حذف نمی‌شود. پس از بازیابی Consumer، پیام دوباره در اختیار آن قرار می‌گیرد.

واژه‌نامه

اصطلاح توضیح
Producer تولیدکننده و ارسال‌کننده پیام
Consumer دریافت‌کننده و پردازش‌کننده پیام
Queue محلی برای نگهداری پیام‌ها تا زمان پردازش
Acknowledgment تأیید موفقیت پردازش پیام توسط Consumer
Idempotency ایمن بودن پردازش تکراری یک پیام

قانون طلایی

🎯
هر عملیاتی که برای پاسخ فوری به کاربر ضروری نیست، می‌تواند گزینه مناسبی برای پردازش Asynchronous از طریق Message Broker باشد.

جمع‌بندی

Message Broker واسطه‌ای برای انتقال پیام میان سرویس‌ها است. با استفاده از آن، Producer و Consumer لازم نیست مستقیماً یکدیگر را بشناسند یا هم‌زمان در دسترس باشند.

این ابزار وابستگی سرویس‌ها را کاهش می‌دهد و تحمل خطا، مقیاس‌پذیری و انعطاف سیستم را افزایش می‌دهد؛ اما نیازمند مدیریت صحیح پیام‌های تکراری، خطاها، مانیتورینگ و سازگاری داده‌ها است.

➡️
در بخش بعد با Message Queue آشنا می‌شویم و بررسی می‌کنیم پیام‌ها چگونه در صف قرار می‌گیرند و میان Consumerها توزیع می‌شوند.

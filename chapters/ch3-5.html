📨 ارتباط Asynchronous

Message-Driven Architecture چیست؟

📅 به‌روزرسانی: 1405/04/25
⏱️ زمان مطالعه: ۷ دقیقه
👤 تیم فنی

Message-Driven Architecture سبکی از معماری نرم‌افزار است که در آن اجزای سیستم از طریق ارسال و دریافت Message با یکدیگر ارتباط برقرار می‌کنند.

به‌جای اینکه یک سرویس مستقیماً سرویس دیگری را فراخوانی کند، Message را داخل Message Broker قرار می‌دهد تا سرویس مقصد آن را در زمان مناسب پردازش کند.

🎯
هدف اصلی Message-Driven Architecture کاهش وابستگی مستقیم میان سرویس‌ها و انجام پردازش‌ها به‌صورت Asynchronous است.

یک مثال ساده

فرض کنید پس از ثبت سفارش باید اطلاعات آن برای پردازش پرداخت ارسال شود.

سرویس Order یک Message با نام ProcessPayment ایجاد کرده و آن را داخل Queue قرار می‌دهد.

Message-Driven Flow 

Order Service

      │

ProcessPayment Message

      ▼

Payment Queue

      │

      ▼

Payment Service

      │

Process Payment

سرویس Order برای اجرای Payment Service منتظر نمی‌ماند و Payment Service هر زمان آماده باشد Message را پردازش می‌کند.

✅
اگر Payment Service موقتاً از دسترس خارج شود، Message می‌تواند داخل Queue باقی بماند و پس از بازیابی سرویس پردازش شود.

اجزای اصلی

جزء وظیفه
Message Producer Message را تولید و ارسال می‌کند.
Message Broker Message را دریافت، نگهداری و مسیریابی می‌کند.
Queue یا Topic محل قرارگیری Message تا زمان پردازش است.
Message Consumer Message را دریافت و عملیات مربوط به آن را اجرا می‌کند.

Message چه اطلاعاتی دارد؟

Message معمولاً شامل نام عملیات، شناسه یکتا، زمان ایجاد و اطلاعات موردنیاز Consumer است.

ProcessPayment Message 

{
  "messageId": "98f31b42",
  "messageType": "ProcessPayment",
  "orderId": 1001,
  "customerId": 25,
  "amount": 850000,
  "createdAt": "2026-07-17T09:30:00Z"
}
💡
Message باید یک قرارداد مستقل داشته باشد و بهتر است Entity کامل Database داخل آن قرار نگیرد.

تفاوت Message-Driven و Event-Driven

این دو معماری شبیه یکدیگر هستند، اما نوع Message و هدف ارتباط در آن‌ها متفاوت است.

ویژگی Message-Driven Event-Driven
هدف ارسال درخواست برای انجام یک کار اعلام وقوع یک اتفاق
نمونه ProcessPayment PaymentCompleted
دریافت‌کننده معمولاً یک Consumer مشخص ممکن است چند Consumer باشد
نوع پیام اغلب Command Event
📌
Message-Driven یک مفهوم گسترده‌تر است و می‌تواند شامل Command، Event یا انواع دیگر Message باشد.

مزایای Message-Driven Architecture

  • کاهش وابستگی مستقیم میان سرویس‌ها
  • انجام پردازش‌های زمان‌بر در پس‌زمینه
  • افزایش تحمل خطا
  • امکان Retry پیام‌های ناموفق
  • افزایش مقیاس‌پذیری با چند Consumer
  • کنترل بهتر افزایش ناگهانی حجم درخواست‌ها

چالش‌ها

چالش توضیح
Duplicate Message ممکن است یک Message چند بار تحویل داده شود.
Message Ordering ترتیب پردازش همیشه تضمین‌شده نیست.
Queue Backlog کندی Consumer باعث تجمع Messageها می‌شود.
Debugging ردیابی عملیات میان چند سرویس دشوارتر است.
Eventual Consistency نتیجه عملیات ممکن است با کمی تأخیر در سیستم دیده شود.

Best Practiceها

انجام دهید ✅ انجام ندهید ❌
برای هر Message شناسه یکتا قرار دهید. پردازش دقیقاً یک‌باره را فرض نکنید.
Consumer را Idempotent طراحی کنید. Message تکراری را بدون بررسی پردازش نکنید.
از Retry محدود و DLQ استفاده کنید. Retry بی‌نهایت انجام ندهید.
Queue Backlog را مانیتور کنید. فقط وضعیت Broker را بررسی نکنید.
قرارداد Message را Version کنید. ساختار Message را ناگهانی تغییر ندهید.

چه زمانی مناسب است؟

  • عملیات برای پاسخ فوری کاربر ضروری نیست.
  • پردازش زمان‌بر یا سنگین است.
  • سرویس مقصد ممکن است موقتاً در دسترس نباشد.
  • حجم درخواست‌ها ناگهان افزایش پیدا می‌کند.
  • نیاز به Retry و پردازش مطمئن وجود دارد.

اشتباه رایج

❌
ارسال Message به Broker به معنی موفقیت کامل عملیات نیست. سیستم باید وضعیت پردازش، Retry، Dead Letter Queue و خطاهای Consumer را نیز مانیتور کند.

نکته مصاحبه

🎓
سؤال: اگر یک Message دو بار به Consumer تحویل داده شود، چگونه از اجرای دوباره عملیات جلوگیری می‌کنید؟

پاسخ: Consumer باید Idempotent باشد. معمولاً شناسه Message در Database ذخیره می‌شود و قبل از پردازش بررسی می‌شود که Message قبلاً پردازش نشده باشد.

قانون طلایی

🎯
در معماری Message-Driven هیچ‌گاه فرض نکنید یک Message فقط یک بار و دقیقاً به همان ترتیب مورد انتظار پردازش می‌شود.

جمع‌بندی

در Message-Driven Architecture، سرویس‌ها از طریق Message و Message Broker با یکدیگر ارتباط برقرار می‌کنند. این روش وابستگی مستقیم را کاهش داده و امکان پردازش Asynchronous، Retry و مقیاس‌پذیری بهتر را فراهم می‌کند.

با این حال، طراحی Consumerهای Idempotent، مدیریت Messageهای ناموفق، Monitoring و کنترل Queue Backlog برای پایداری سیستم ضروری است.

➡️
در بخش بعد تفاوت Message Queue و Publish/Subscribe را بررسی می‌کنیم و با نحوه توزیع Message میان چند Consumer آشنا می‌شویم.

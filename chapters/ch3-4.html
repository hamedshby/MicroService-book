📨 ارتباط Asynchronous

Event-Driven Architecture چیست؟

📅 به‌روزرسانی: 1405/04/26
⏱️ زمان مطالعه: ۸ دقیقه
👤 تیم فنی

Event-Driven Architecture یا معماری رویدادمحور، سبکی از معماری نرم‌افزار است که در آن سرویس‌ها با انتشار و دریافت Event با یکدیگر همکاری می‌کنند.

هر سرویس پس از وقوع یک اتفاق مهم، Event مربوط به آن را منتشر می‌کند و سایر سرویس‌ها در صورت نیاز به آن واکنش نشان می‌دهند.

🎯
در معماری Event-Driven، سرویس منتشرکننده لازم نیست بداند چه سرویس‌هایی Event را دریافت می‌کنند یا چه عملیاتی انجام می‌دهند.

یک مثال ساده

فرض کنید کاربر سفارشی را با موفقیت ثبت می‌کند.

سرویس Order رویداد OrderCreated را منتشر می‌کند و چند سرویس مختلف به آن واکنش نشان می‌دهند.

Event-Driven Flow 

Order Service

      │

OrderCreated Event

      ▼

Message Broker

      │

      ├──► Inventory Service
      │       Reserve Products
      │
      ├──► Notification Service
      │       Send Notification
      │
      └──► Analytics Service
              Record Order

در این مدل، اضافه شدن یک Consumer جدید نیازی به تغییر در Order Service ندارد.

✅
هر سرویس فقط مسئول واکنش خود به Event است و منطق سایر سرویس‌ها را نمی‌شناسد.

اجزای اصلی معماری Event-Driven

جزء وظیفه
Event Producer پس از وقوع یک اتفاق، Event را منتشر می‌کند.
Event Broker Event را دریافت و به Consumerهای مناسب منتقل می‌کند.
Event Consumer Event را دریافت و عملیات مربوط به خود را اجرا می‌کند.
Event Contract ساختار و اطلاعات موجود در Event را مشخص می‌کند.

تفاوت معماری Sync و Event-Driven

در ارتباط Sync، سرویس درخواست‌دهنده مستقیماً سرویس مقصد را فراخوانی می‌کند و منتظر پاسخ می‌ماند.

Synchronous 

Order Service

      │

REST Request

      ▼

Inventory Service

      │

Response

      ▼

Order Service Continues

اما در معماری Event-Driven، سرویس Producer فقط Event را منتشر می‌کند و منتظر اجرای Consumerها نمی‌ماند.

Event-Driven 

Order Service

      │

Publish Event

      ▼

Message Broker

      │

Order Service Continues

      │

Consumers Process Later
ویژگی Sync Event-Driven
نوع وابستگی مستقیم غیرمستقیم
انتظار برای پاسخ دارد معمولاً ندارد
تحمل خطا کمتر بیشتر
پیچیدگی ردیابی کمتر بیشتر

Event Notification

در این روش، Event فقط اعلام می‌کند که اتفاقی رخ داده است و معمولاً اطلاعات کمی همراه خود دارد.

Event Notification 

{
  "eventType": "OrderCreated",
  "orderId": 1001
}

Consumer پس از دریافت Event، برای دریافت اطلاعات بیشتر سرویس Producer را فراخوانی می‌کند.

Notification Flow 

Consumer

   │

Receive OrderCreated

   │

GET /orders/1001

   ▼

Order Service
💡
Event Notification پیام‌های کوچک‌تری ایجاد می‌کند، اما ممکن است باعث افزایش درخواست‌های Sync میان سرویس‌ها شود.

Event-Carried State Transfer

در این روش، اطلاعات موردنیاز Consumer داخل خود Event قرار می‌گیرد.

Event-Carried State Transfer 

{
  "eventType": "OrderCreated",
  "orderId": 1001,
  "customerId": 25,
  "customerEmail": "user@example.com",
  "totalAmount": 850000
}

Consumer بدون فراخوانی مجدد Producer می‌تواند عملیات خود را انجام دهد.

⚠️
قرار دادن اطلاعات زیاد در Event باعث افزایش حجم پیام و وابستگی Consumerها به قرارداد Event می‌شود.

Eventual Consistency

در معماری Event-Driven، همه سرویس‌ها دقیقاً در یک لحظه به‌روزرسانی نمی‌شوند.

ممکن است Order ثبت شده باشد، اما چند ثانیه طول بکشد تا Inventory یا Analytics نیز اطلاعات خود را به‌روزرسانی کنند.

Eventual Consistency 

Order Created

      │

      ▼

Event Published

      │

      ├── Inventory Updated after 1s
      │
      ├── Notification Sent after 2s
      │
      └── Analytics Updated after 4s
📌
Eventual Consistency یعنی داده‌ها فوراً یکسان نیستند، اما پس از مدت کوتاهی به وضعیت هماهنگ می‌رسند.

مزایای Event-Driven Architecture

  • کاهش Coupling میان Microserviceها
  • افزایش مقیاس‌پذیری سیستم
  • امکان اضافه شدن Consumer جدید بدون تغییر Producer
  • افزایش تحمل خطا در زمان قطعی موقت سرویس‌ها
  • مناسب برای پردازش‌های Asynchronous و زمان‌بر
  • تقسیم بهتر مسئولیت‌ها میان سرویس‌ها

چالش‌های Event-Driven Architecture

چالش توضیح
Eventual Consistency داده‌ها با کمی تأخیر میان سرویس‌ها هماهنگ می‌شوند.
Duplicate Event ممکن است یک Event بیش از یک بار تحویل داده شود.
Event Ordering ترتیب پردازش Eventها همیشه تضمین‌شده نیست.
Debugging ردیابی جریان عملیات در چند سرویس دشوارتر است.
Schema Evolution تغییر قرارداد Event باید با Consumerها سازگار باشد.
Transaction هماهنگی Database و انتشار Event نیازمند طراحی مناسب است.
⚠️
ذخیره موفق اطلاعات در Database و شکست انتشار Event می‌تواند سیستم را ناسازگار کند. الگوی Transactional Outbox یکی از راهکارهای این مشکل است.

چه زمانی مناسب است؟

  • پس از وقوع یک رویداد، چند سرویس باید واکنش نشان دهند.
  • پردازش‌ها نیازمند پاسخ فوری نیستند.
  • سیستم باید در برابر قطعی موقت سرویس‌ها مقاوم باشد.
  • اضافه شدن Consumerهای جدید در آینده محتمل است.
  • پردازش حجم بالای Event موردنیاز است.

چه زمانی مناسب نیست؟

  • کاربر به پاسخ فوری و قطعی نیاز دارد.
  • عملیات بسیار ساده و کوتاه است.
  • Consistency لحظه‌ای برای عملیات ضروری است.
  • پیچیدگی Message Broker برای سیستم توجیه ندارد.

Best Practiceها

انجام دهید ✅ انجام ندهید ❌
Eventها را بر اساس اتفاقات Business نام‌گذاری کنید. از نام‌های فنی و مبهم استفاده نکنید.
Consumerها را Idempotent طراحی کنید. تحویل دقیقاً یک‌باره را فرض نکنید.
برای Eventها Version در نظر بگیرید. قرارداد Event را ناگهانی تغییر ندهید.
از Retry و Dead Letter Queue استفاده کنید. پیام‌های ناموفق را نادیده نگیرید.
از Distributed Tracing استفاده کنید. فقط Log هر سرویس را جداگانه بررسی نکنید.

اشتباه رایج

❌
تبدیل تمام ارتباط‌های سیستم به Event یک طراحی مناسب نیست. عملیات‌هایی که به پاسخ فوری نیاز دارند همچنان می‌توانند با REST یا gRPC انجام شوند.

نکته مصاحبه

🎓
سؤال: مهم‌ترین تفاوت معماری Event-Driven با ارتباط Synchronous چیست؟

پاسخ: در ارتباط Sync، سرویس درخواست‌دهنده مستقیماً به سرویس مقصد وابسته است و منتظر پاسخ می‌ماند؛ اما در معماری Event-Driven، Producer فقط Event را منتشر می‌کند و Consumerها به‌صورت مستقل و معمولاً Asynchronous آن را پردازش می‌کنند.

قانون طلایی

🎯
Event-Driven Architecture برای کاهش وابستگی طراحی می‌شود؛ نه برای انتقال وابستگی مستقیم سرویس‌ها به Message Broker.

جمع‌بندی

Event-Driven Architecture معماری‌ای است که در آن سرویس‌ها با انتشار و دریافت Event با یکدیگر همکاری می‌کنند. این روش Coupling را کاهش می‌دهد و مقیاس‌پذیری و تحمل خطا را افزایش می‌دهد.

در مقابل، چالش‌هایی مانند Eventual Consistency، پیام‌های تکراری، ترتیب Eventها، تغییر قرارداد و Distributed Tracing باید در طراحی سیستم مدیریت شوند.

➡️
در بخش بعد با Message-Driven Architecture آشنا می‌شویم و تفاوت آن را با Event-Driven Architecture بررسی می‌کنیم.

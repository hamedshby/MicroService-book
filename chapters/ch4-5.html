Queue در RabbitMQ

📅 به‌روزرسانی: 1405/04/30
⏱️ زمان مطالعه: ۱۰ دقیقه
👤 تیم فنی

Queue محلی برای نگهداری پیام‌ها در RabbitMQ است. پیام پس از عبور از Exchange وارد Queue می‌شود و تا زمان دریافت توسط Consumer در آن باقی می‌ماند.

در سناریوی ما، Order Service پیام OrderCreated را منتشر می‌کند و Notification Service آن را از Queue دریافت می‌کند.

Message Flow 

Order Service

      │

OrderCreated

      ▼

Order Exchange

      │

      ▼

order.notification.queue

      │

      ▼

Notification Service
🎯
Queue میان Producer و Consumer قرار می‌گیرد و اختلاف سرعت آن‌ها را مدیریت می‌کند.

تعریف Queue در .NET

برای ساخت Queue از متد QueueDeclareAsync استفاده می‌کنیم.

C# - Declare Queue 
using RabbitMQ.Client;

var factory = new ConnectionFactory
{
    HostName = "localhost",
    UserName = "guest",
    Password = "guest",
    ClientProvidedName = "OrderService"
};

await using var connection =
    await factory.CreateConnectionAsync();

await using var channel =
    await connection.CreateChannelAsync();

const string queueName = "order.notification.queue";

await channel.QueueDeclareAsync(
    queue: queueName,
    durable: true,
    exclusive: false,
    autoDelete: false,
    arguments: null);

اگر Queue از قبل وجود نداشته باشد ساخته می‌شود. اگر وجود داشته باشد، RabbitMQ ویژگی‌های آن را بررسی می‌کند.

⚠️
نمی‌توان یک Queue موجود را دوباره با ویژگی‌های متفاوت تعریف کرد. برای مثال، Queue ساخته‌شده با durable برابر true را نمی‌توان با durable برابر false تعریف کرد.

ویژگی Durable

ویژگی durable مشخص می‌کند تعریف Queue پس از Restart شدن RabbitMQ باقی بماند یا خیر.

C# - Durable Queue 
await channel.QueueDeclareAsync(
    queue: "order.notification.queue",
    durable: true,
    exclusive: false,
    autoDelete: false,
    arguments: null);
مقدار رفتار
true تعریف Queue پس از Restart باقی می‌ماند.
false Queue موقتی است و پس از Restart باقی نمی‌ماند.
📌
Durable بودن Queue به‌تنهایی برای حفظ پیام‌ها کافی نیست. پیام نیز باید به صورت Persistent منتشر شود.

ارسال پیام Persistent

برای افزایش ماندگاری پیام OrderCreated، ویژگی Persistent را هنگام انتشار فعال می‌کنیم.

C# - Persistent Message 
using System.Text;
using System.Text.Json;
using RabbitMQ.Client;

var message = new
{
    MessageId = Guid.NewGuid(),
    MessageType = "OrderCreated",
    OrderId = 1001,
    CustomerId = 25,
    TotalAmount = 850000,
    CreatedAt = DateTime.UtcNow
};

byte[] body = Encoding.UTF8.GetBytes(
    JsonSerializer.Serialize(message));

var properties = new BasicProperties
{
    Persistent = true,
    ContentType = "application/json",
    MessageId = message.MessageId.ToString(),
    Type = message.MessageType
};

await channel.BasicPublishAsync(
    exchange: "order.direct",
    routingKey: "order.created",
    mandatory: true,
    basicProperties: properties,
    body: body);
✅
برای پیام‌های مهم، Queue را Durable و پیام را Persistent تعریف کنید.

ویژگی Exclusive

اگر exclusive برابر true باشد، Queue فقط توسط همان Connection قابل استفاده است و با بسته شدن Connection حذف می‌شود.

C# - Exclusive Queue 
QueueDeclareOk result =
    await channel.QueueDeclareAsync(
        queue: string.Empty,
        durable: false,
        exclusive: true,
        autoDelete: true,
        arguments: null);

string generatedQueueName = result.QueueName;

با ارسال نام خالی، RabbitMQ یک نام یکتا برای Queue تولید می‌کند.

💡
Exclusive Queue بیشتر برای سناریوهای موقتی مانند Reply Queue، دریافت Log یا Subscriptionهای کوتاه‌مدت مناسب است.

ویژگی Auto Delete

اگر autoDelete برابر true باشد، Queue پس از قطع آخرین Consumer به‌صورت خودکار حذف می‌شود.

C# - Auto Delete Queue 
await channel.QueueDeclareAsync(
    queue: "order.monitoring.temporary",
    durable: false,
    exclusive: false,
    autoDelete: true,
    arguments: null);
⚠️
برای Queueهای اصلی مانند order.notification.queue معمولاً autoDelete باید false باشد.

مقایسه ویژگی‌های Queue

ویژگی وظیفه پیشنهاد برای Order Service
durable حفظ تعریف Queue پس از Restart true
exclusive محدود کردن Queue به یک Connection false
autoDelete حذف Queue پس از قطع Consumerها false

اتصال Queue به Exchange

پس از تعریف Queue باید آن را با یک Binding به Exchange متصل کنیم.

C# - Queue Binding 
const string exchangeName = "order.direct";
const string queueName = "order.notification.queue";
const string routingKey = "order.created";

await channel.ExchangeDeclareAsync(
    exchange: exchangeName,
    type: ExchangeType.Direct,
    durable: true,
    autoDelete: false);

await channel.QueueDeclareAsync(
    queue: queueName,
    durable: true,
    exclusive: false,
    autoDelete: false,
    arguments: null);

await channel.QueueBindAsync(
    queue: queueName,
    exchange: exchangeName,
    routingKey: routingKey);

اکنون پیام‌هایی با Routing Key برابر order.created وارد Notification Queue می‌شوند.

دریافت پیام از Queue

Notification Service پیام را دریافت می‌کند و پس از پردازش موفق برای آن Acknowledgment ارسال می‌کند.

C# - Queue Consumer 
using System.Text;
using RabbitMQ.Client.Events;

const string queueName = "order.notification.queue";

await channel.BasicQosAsync(
    prefetchSize: 0,
    prefetchCount: 10,
    global: false);

var consumer = new AsyncEventingBasicConsumer(channel);

consumer.ReceivedAsync += async (_, eventArgs) =>
{
    try
    {
        string json = Encoding.UTF8.GetString(
            eventArgs.Body.Span);

        Console.WriteLine(
            $"OrderCreated received: {json}");

        // ارسال پیام تأیید سفارش برای مشتری
        await SendOrderNotificationAsync(json);

        await channel.BasicAckAsync(
            deliveryTag: eventArgs.DeliveryTag,
            multiple: false);
    }
    catch (Exception exception)
    {
        Console.WriteLine(exception.Message);

        await channel.BasicNackAsync(
            deliveryTag: eventArgs.DeliveryTag,
            multiple: false,
            requeue: false);
    }
};

await channel.BasicConsumeAsync(
    queue: queueName,
    autoAck: false,
    consumer: consumer);
📌
با autoAck برابر false، پیام فقط پس از اجرای BasicAckAsync از Queue حذف می‌شود.

Prefetch Count

ویژگی Prefetch Count مشخص می‌کند RabbitMQ چند پیام تأییدنشده را هم‌زمان برای هر Consumer ارسال کند.

C# - Prefetch 
await channel.BasicQosAsync(
    prefetchSize: 0,
    prefetchCount: 10,
    global: false);

در این مثال، Notification Service حداکثر ۱۰ پیام تأییدنشده را هم‌زمان دریافت می‌کند.

⚠️
مقدار بسیار زیاد Prefetch می‌تواند باعث توزیع نامتعادل پیام‌ها و افزایش مصرف حافظه Consumer شود.

Queueهای ثابت و موقت

نوع Queue تنظیمات معمول کاربرد
Queue ثابت Durable و Non-Exclusive پردازش OrderCreated
Queue موقت Non-Durable و Exclusive Subscription یا Reply Queue
Auto-Delete Queue AutoDelete برابر true مصرف‌کننده‌های کوتاه‌مدت

اشتباه رایج

❌
Durable بودن Queue به معنی ماندگاری تمام پیام‌ها نیست. برای پیام‌های مهم باید Message Persistence، Acknowledgment و Publisher Confirm نیز به‌درستی تنظیم شوند.

نکته مصاحبه

🎓
سؤال: تفاوت Durable و Persistent چیست؟

پاسخ: Durable ویژگی Queue است و باعث حفظ تعریف Queue پس از Restart می‌شود؛ اما Persistent ویژگی Message است و برای افزایش ماندگاری خود پیام استفاده می‌شود.

قانون طلایی

🎯
برای Queueهای اصلی سیستم معمولاً از durable: true، exclusive: false و autoDelete: false استفاده کنید؛ اما تنظیم نهایی باید بر اساس طول عمر و کاربرد Queue انتخاب شود.

جمع‌بندی

Queue پیام‌ها را تا زمان پردازش توسط Consumer نگهداری می‌کند. ویژگی‌های Durable، Exclusive و Auto Delete مشخص می‌کنند Queue چه طول عمری داشته باشد و چگونه حذف شود.

در سناریوی Order Service، پیام OrderCreated داخل order.notification.queue قرار می‌گیرد و Notification Service آن را با Acknowledgment دستی پردازش می‌کند.

➡️
در بخش بعد با Binding و Routing Key آشنا می‌شویم و نحوه اتصال Exchange به Queue را با مثال‌های .NET بررسی می‌کنیم.

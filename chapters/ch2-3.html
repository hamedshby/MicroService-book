``` GraphQL یک زبان Query و یک Runtime برای طراحی و پیاده‌سازی API است که به Client اجازه می‌دهد دقیقاً مشخص کند چه داده‌هایی را نیاز دارد. ```

``` برخلاف REST که معمولاً برای هر Resource یک یا چند Endpoint در نظر گرفته می‌شود، در GraphQL بیشتر درخواست‌ها از طریق یک Endpoint واحد ارسال می‌شوند. ```

🎯
``` هدف اصلی GraphQL این است که Client فقط داده‌های موردنیاز خود را دریافت کند و کنترل بیشتری روی ساختار Response داشته باشد.
```

چرا GraphQL مهم است؟

``` در سیستم‌های مدرن، یک Client ممکن است برای نمایش یک صفحه به اطلاعات چندین Resource مختلف نیاز داشته باشد. ```

``` برای مثال، صفحه جزئیات یک سفارش ممکن است شامل اطلاعات سفارش، کاربر، پرداخت، محصولات و وضعیت ارسال باشد. ```

``` در معماری REST، Client ممکن است مجبور شود چندین درخواست مختلف ارسال کند. ```

```
REST Requests 

```

GET /api/orders/1001

GET /api/users/25

GET /api/payments/order/1001

GET /api/products/501

GET /api/shipping/order/1001

``` در GraphQL، Client می‌تواند تمام داده‌های موردنیاز خود را در قالب یک Query درخواست کند. ```

```

``` Schema مشخص می‌کند چه Typeهایی وجود دارند، چه فیلدهایی قابل دریافت هستند و چه عملیات‌هایی توسط Client قابل اجرا است. ```

``` GraphQL Server ابتدا Query دریافتی را با Schema مقایسه می‌کند. اگر Query معتبر باشد، Resolver مربوط به هر Field اجرا می‌شود. ```

``` در نهایت، Response دقیقاً با همان ساختاری که Client درخواست کرده است بازگردانده می‌شود. ```

💡
``` در GraphQL، Client ساختار Response را مشخص می‌کند؛ اما فقط می‌تواند فیلدهایی را درخواست کند که در Schema تعریف شده‌اند.
```

Schema

``` Schema قرارداد اصلی میان Client و Server در GraphQL است. ```

``` این قرارداد معمولاً با استفاده از Schema Definition Language یا SDL نوشته می‌شود. ```

```
GraphQL Schema 

```

type Product {
id: ID!
title: String!
price: Float!
}

type OrderItem {
quantity: Int!
product: Product!
}

type Order {
id: ID!
status: String!
items: [OrderItem!]!
}

type Query {
order(id: ID!): Order
}

``` علامت ! به این معنا است که مقدار آن Field نمی‌تواند Null باشد. ```

``` همچنین ساختار [OrderItem!]! به این معنا است که لیست نمی‌تواند Null باشد و اعضای داخل آن نیز نباید Null باشند. ```

⚠️
``` تغییر Schema باید با دقت انجام شود؛ زیرا حذف یا تغییر نوع یک Field ممکن است Clientهای موجود را دچار مشکل کند.
```

Mutation

``` از Mutation برای ایجاد، تغییر یا حذف اطلاعات استفاده می‌شود. ```

``` Mutation مشابه عملیات‌های POST، PUT، PATCH و DELETE در REST است. ```

```
Mutation Example 

```

mutation CreateOrder {

``` Mutation می‌تواند علاوه بر اجرای عملیات، اطلاعات نتیجه را نیز در همان Response برگرداند. ```

📌
``` بهتر است ورودی Mutationها در Typeهای جداگانه مانند CreateOrderInput تعریف شوند تا Schema خواناتر و قابل توسعه‌تر باقی بماند.
```

Subscription

``` Subscription برای دریافت تغییرات Real-time از Server استفاده می‌شود. ```

``` برخلاف Query و Mutation که معمولاً یک Request و یک Response دارند، Subscription یک ارتباط طولانی‌مدت ایجاد می‌کند. ```

```

```

▼

│

│

▼

Another Event Sent
```
Subscription Example 

```

subscription {

orderId
status
updatedAt
}
}

``` Subscriptionها معمولاً برای اعلان‌ها، چت، داشبوردهای زنده و نمایش وضعیت Real-time عملیات استفاده می‌شوند. ```

⚠️
``` Subscription معمولاً به WebSocket یا یک ارتباط پایدار نیاز دارد و مدیریت مقیاس‌پذیری آن نسبت به Query و Mutation پیچیده‌تر است.
```

``` هر Resolver می‌تواند داده را از Database، Cache، Microservice یا یک API خارجی دریافت کند. ```

``` برای مثال، Resolver مربوط به Field سفارش ممکن است ابتدا سفارش را از Database دریافت کند و Resolverهای داخلی آن اطلاعات کاربر، پرداخت و ارسال را از سرویس‌های دیگر دریافت کنند. ```

💡
``` Resolverها نباید به منطق تجاری حجیم تبدیل شوند. بهتر است منطق اصلی در Application Service یا Domain Service قرار گیرد و Resolver فقط نقش هماهنگ‌کننده داشته باشد.
```

Over-fetching

``` Over-fetching زمانی رخ می‌دهد که Server اطلاعات بیشتری از نیاز Client برگرداند. ```

``` برای مثال، ممکن است Client فقط نام محصول را نیاز داشته باشد اما REST API تمام اطلاعات محصول را ارسال کند. ```

```
REST Response 

```

{
"id": 501,
"title": "Keyboard",

"price": 120,
"weight": 0.9,
"categoryId": 12,
"createdAt": "2026-07-10",
"updatedAt": "2026-07-15",
"warehouseId": 4
}

``` در حالی که Client فقط به این اطلاعات نیاز دارد: ```

```

```

}
}
💡
``` GraphQL با انتخاب Fieldها توسط Client، مشکل Over-fetching را تا حد زیادی کاهش می‌دهد.
```

Query Depth

``` Client در GraphQL می‌تواند Queryهای تو در تو ارسال کند. ```

``` اگر محدودیتی روی عمق Query وجود نداشته باشد، ممکن است یک Query بسیار سنگین ایجاد شود. ```

```

```

}

}
}
}

}
}
}

❌
``` اجازه دادن به Queryهای بسیار عمیق می‌تواند باعث مصرف زیاد CPU، Memory، Database Connection و پهنای باند شود.
```

``` برای کنترل این مشکل می‌توان از Depth Limit، Complexity Limit و Timeout استفاده کرد. ```

Query Complexity

``` عمق Query تنها معیار سنگین بودن آن نیست. ```

``` ممکن است یک Query عمق کمی داشته باشد اما تعداد بسیار زیادی آیتم درخواست کند. ```

```

```

id

product {

}

``` در چنین شرایطی باید برای هر Field هزینه مشخصی در نظر گرفته شود و Queryهایی که از حد مجاز عبور می‌کنند رد شوند. ```

📌
``` Pagination، محدودیت تعداد آیتم‌ها و Query Cost Analysis از مهم‌ترین ابزارهای محافظت از GraphQL API هستند.
```

Error Handling

``` نحوه مدیریت خطا در GraphQL با REST تفاوت دارد. ```

``` در بسیاری از پیاده‌سازی‌ها، حتی در صورت رخ دادن خطا ممکن است Response با Status Code برابر 200 بازگردد و جزئیات خطا در بخش errors قرار گیرد. ```

```
GraphQL Error Response 

```

{

"order": null
},
"errors": [
{
"message": "Order not found",
"path": [
"order"
],
"extensions": {
"code": "ORDER_NOT_FOUND"
}

``` GraphQL همچنین می‌تواند Partial Response برگرداند؛ یعنی بخشی از اطلاعات موفق باشد و بخشی دیگر با خطا مواجه شود. ```

💡
``` Client باید علاوه بر بخش data، همیشه بخش errors را نیز بررسی کند.
```

Caching در GraphQL

``` Caching در REST معمولاً با URL و HTTP Method ساده‌تر انجام می‌شود. ```

``` در GraphQL، بیشتر درخواست‌ها به یک Endpoint مشترک ارسال می‌شوند و Query در Body درخواست قرار دارد؛ بنابراین Caching نیازمند طراحی دقیق‌تری است. ```

``` روش‌های رایج Caching عبارت‌اند از: ```

    ```
  • Cache کردن داده داخل Resolver
  • استفاده از DataLoader Cache
  • استفاده از Distributed Cache مانند Redis
  • Persisted Query
  • Client-side Normalized Cache
    • ```
📌
``` Cache باید با توجه به سطح دسترسی کاربران طراحی شود؛ زیرا Cache کردن Response بدون در نظر گرفتن Identity ممکن است اطلاعات یک کاربر را در اختیار کاربر دیگری قرار دهد.
```

✅
``` Persisted Query برای اپلیکیشن‌های موبایل، APIهای عمومی و سیستم‌هایی که به کنترل امنیتی بیشتری نیاز دارند بسیار مفید است.
```

```

مزایای GraphQL

``` ```
مزیت توضیح
کاهش Over-fetching Client فقط Fieldهای موردنیاز را دریافت می‌کند.
کاهش Under-fetching داده‌های مرتبط را می‌توان در یک Query دریافت کرد.
Schema قوی ساختار API و Typeها به‌صورت واضح تعریف می‌شوند.
توسعه سریع Frontend Client برای دریافت ترکیب‌های جدید داده کمتر به Endpoint جدید نیاز دارد.
ابزارهای توسعه مناسب امکان Auto-complete، Documentation و بررسی Typeها وجود دارد.
API یکپارچه چند Microservice می‌توانند پشت یک GraphQL Gateway قرار بگیرند.
Partial Response در صورت خطای بخشی از Query، سایر داده‌های موفق قابل بازگشت هستند.

معایب GraphQL

``` ```
چالش توضیح
پیچیدگی پیاده‌سازی طراحی Schema، Resolver، Authorization و Error Handling نیازمند دقت است.
N+1 Query Resolverهای نامناسب ممکن است تعداد زیادی درخواست داخلی ایجاد کنند.
Caching دشوارتر Cache مبتنی بر URL مانند REST به‌سادگی قابل استفاده نیست.
Queryهای پرهزینه Client می‌تواند Queryهای عمیق یا بزرگ ارسال کند.
Monitoring پیچیده‌تر تمام عملیات‌ها ممکن است از یک Endpoint عبور کنند.
File Upload آپلود فایل معمولاً به طراحی و قرارداد جداگانه نیاز دارد.
پیچیدگی Federation ترکیب چند Schema نیازمند حاکمیت و هماهنگی میان تیم‌ها است.

``` ```

Under-fetching احتمال بیشتر احتمال کمتر

💡
``` GraphQL جایگزین مطلق REST نیست. انتخاب میان REST و GraphQL باید بر اساس نوع Client، پیچیدگی داده، نیازهای تیم و معماری سیستم انجام شود.
```

Schema را بر اساس Domain طراحی کنید. Schema را مستقیماً از جدول‌های Database تولید نکنید.
برای لیست‌ها Pagination قرار دهید. لیست‌های نامحدود برنگردانید.
از DataLoader برای Batch و Cache استفاده کنید. برای هر آیتم یک Query جداگانه اجرا نکنید.
Query Depth و Complexity را محدود کنید. هر Query دلخواهی را بدون محدودیت اجرا نکنید.
Authorization را در سمت Server اعمال کنید. به مخفی بودن Field در UI اعتماد نکنید.
برای خطاها Error Code مشخص تعریف کنید. جزئیات داخلی Exception را به Client نمایش ندهید.
Resolverها را ساده و سبک نگه دارید. تمام Business Logic را داخل Resolver قرار ندهید.
Fieldهای قدیمی را Deprecate کنید. Fieldهای پرکاربرد را ناگهانی حذف نکنید.
برای Production از Observability استفاده کنید. فقط Endpoint کلی GraphQL را مانیتور نکنید.

اشتباهات رایج

``` اشتباه رایج اول: تبدیل مستقیم تمام Entityهای Database به GraphQL Type. این کار باعث نشت ساختار Database به API و افزایش Coupling می‌شود.
```
❌
``` اشتباه رایج دوم: اجرای مستقیم Query پایگاه داده داخل هر Resolver بدون استفاده از Batch، Projection یا DataLoader.
```
❌
``` اشتباه رایج سوم: فعال کردن Queryهای نامحدود بدون Depth Limit، Complexity Limit، Pagination و Timeout.
```
❌
``` اشتباه رایج چهارم: استفاده از GraphQL صرفاً به دلیل محبوب بودن آن، بدون وجود مشکل واقعی در REST API فعلی.
```

نکته مصاحبه

🎓
``` سؤال رایج مصاحبه: مشکل N+1 در GraphQL چیست و چگونه آن را حل می‌کنید؟

پاسخ: زمانی که ابتدا یک لیست دریافت می‌شود و سپس برای هر آیتم یک Query یا Request جداگانه اجرا می‌شود، مشکل N+1 رخ می‌دهد. برای حل آن می‌توان از DataLoader، Batch Query، Cache، Projection و دریافت گروهی داده‌ها استفاده کرد.
```
🎓
``` سؤال رایج مصاحبه: آیا GraphQL همیشه بهتر از REST است؟

پاسخ: خیر. GraphQL برای سناریوهایی که Clientهای مختلف به ساختارهای متفاوتی از داده نیاز دارند بسیار مناسب است؛ اما REST برای APIهای ساده، عملیات‌های فایل، Caching مبتنی بر HTTP و سیستم‌های کم‌پیچیدگی می‌تواند انتخاب بهتری باشد.
```

چه زمانی GraphQL مناسب است؟

    ```
  • چند Client مختلف مانند Web، Mobile و Tablet وجود دارد.
  • هر Client به مجموعه متفاوتی از داده‌ها نیاز دارد.
  • صفحات دارای داده‌های تو در تو و مرتبط هستند.
  • تعداد زیاد REST Request باعث کاهش Performance شده است.
  • تیم Frontend به انعطاف بیشتری برای دریافت داده نیاز دارد.
  • قرار است چند Microservice پشت یک API یکپارچه قرار گیرند.
  • نیاز به Schema و قرارداد Type-safe وجود دارد.
    • ```

چه زمانی GraphQL مناسب نیست؟

    ```
  • API بسیار ساده و دارای چند Endpoint محدود است.
  • بیشتر عملیات‌ها مربوط به Upload و Download فایل هستند.
  • HTTP Caching اهمیت بسیار زیادی دارد.
  • تیم تجربه کافی برای مدیریت پیچیدگی GraphQL ندارد.
  • ساختار Response همیشه ثابت و از قبل مشخص است.
  • نیازی به Queryهای انعطاف‌پذیر از سمت Client وجود ندارد.
  • محدودیت زمانی پروژه اجازه طراحی صحیح Schema را نمی‌دهد.
    • ```
⚠️
``` استفاده از GraphQL بدون نیاز واقعی می‌تواند به جای حل مسئله، پیچیدگی جدیدی به سیستم اضافه کند.
```

جمع‌بندی

``` GraphQL یک روش قدرتمند برای طراحی APIهای انعطاف‌پذیر است که به Client اجازه می‌دهد دقیقاً داده‌های موردنیاز خود را درخواست کند. ```

``` این فناوری می‌تواند مشکلات Over-fetching و Under-fetching را کاهش دهد و یک API یکپارچه روی چند Microservice ایجاد کند. ```

``` با این حال، GraphQL پیچیدگی‌هایی مانند N+1 Query، Queryهای پرهزینه، Caching، Authorization، Monitoring و مدیریت Schema را نیز به همراه دارد. ```

``` موفقیت در استفاده از GraphQL به طراحی صحیح Schema، Resolverهای بهینه، محدودیت Query، Pagination، DataLoader و Observability مناسب وابسته است. ```

🎯
``` GraphQL یک ابزار قدرتمند است، اما مانند هر ابزار معماری باید بر اساس مسئله واقعی سیستم انتخاب شود. در بسیاری از پروژه‌ها، ترکیب REST، GraphQL و ارتباط‌های Asynchronous بهترین نتیجه را ایجاد می‌کند.
```

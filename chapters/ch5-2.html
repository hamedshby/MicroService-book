معماری Apache Kafka

📅 به‌روزرسانی: 1405/04/31
⏱️ زمان مطالعه: ۱۰ دقیقه
👤 تیم فنی

Kafka از یک معماری Distributed استفاده می‌کند که در آن Eventها داخل Topic ذخیره شده و میان چند Broker توزیع می‌شوند.

Producerها Eventها را منتشر می‌کنند و Consumerها آن‌ها را براساس Offset می‌خوانند.

🎯
هدف معماری Kafka، پردازش حجم بالای Event با مقیاس‌پذیری، تحمل خطا و امکان خواندن مجدد داده‌ها است.

Kafka Cluster از چند Broker تشکیل می‌شود. هر Broker بخشی از Partitionهای Topicها را نگهداری می‌کند.

Broker چیست؟

هر سرور Kafka یک Broker نامیده می‌شود.

Broker مسئول دریافت Event از Producer، ذخیره آن و ارائه Event به Consumerها است.

Kafka Cluster 

Kafka Cluster

   ├── Broker 1
   │     ├── Topic A - Partition 0
   │     └── Topic B - Partition 1
   │
   ├── Broker 2
   │     ├── Topic A - Partition 1
   │     └── Topic B - Partition 0
   │
   └── Broker 3
         └── Topic A - Partition 2
✅
اضافه کردن Brokerهای بیشتر می‌تواند ظرفیت ذخیره‌سازی و توان پردازش Cluster را افزایش دهد.

Topic چیست؟

Topic یک نام منطقی برای دسته‌بندی Eventهای مرتبط است.

برای مثال، Eventهای مربوط به پرداخت می‌توانند داخل Topic زیر قرار بگیرند:

Topic Name 
payment-completed

Producer پیام را داخل Topic منتشر می‌کند و Consumer به همان Topic Subscribe می‌شود.

Partition چیست؟

هر Topic می‌تواند به چند Partition تقسیم شود.

Partitionها امکان ذخیره و پردازش موازی Eventها را فراهم می‌کنند.

Topic Partitions 

Topic: payment-completed

   ├── Partition 0
   │     ├── Offset 0
   │     ├── Offset 1
   │     └── Offset 2
   │
   ├── Partition 1
   │     ├── Offset 0
   │     └── Offset 1
   │
   └── Partition 2
         ├── Offset 0
         └── Offset 1
💡
ترتیب Eventها فقط داخل یک Partition تضمین می‌شود، نه در کل Topic.

Message Key و انتخاب Partition

Producer می‌تواند هنگام ارسال Event یک Key مشخص کند.

Kafka معمولاً Eventهایی با Key یکسان را داخل یک Partition قرار می‌دهد.

Partitioning by Key 

OrderId = 1001

PaymentStarted
PaymentCompleted
PaymentRefunded

        │

Same Key: 1001

        ▼

Partition 2

این کار باعث می‌شود ترتیب Eventهای مربوط به یک Order حفظ شود.

⚠️
انتخاب Key نامناسب ممکن است باعث شود بیشتر Eventها وارد یک Partition شوند و Load به‌صورت نامتوازن توزیع شود.

ارسال Event با Key در .NET

C# Producer 
using Confluent.Kafka;
using System.Text.Json;

var config = new ProducerConfig
{
    BootstrapServers = "localhost:9092",
    Acks = Acks.All
};

using var producer =
    new ProducerBuilder<string, string>(config).Build();

var paymentCompleted = new
{
    PaymentId = 1205,
    OrderId = 1001,
    Amount = 850000,
    OccurredAt = DateTime.UtcNow
};

var message = new Message<string, string>
{
    Key = paymentCompleted.OrderId.ToString(),
    Value = JsonSerializer.Serialize(paymentCompleted)
};

var result = await producer.ProduceAsync(
    "payment-completed",
    message
);

Console.WriteLine(
    $"Partition: {result.Partition}, Offset: {result.Offset}"
);

در این مثال، OrderId به‌عنوان Key استفاده شده است تا Eventهای یک سفارش معمولاً در یک Partition قرار بگیرند.

Offset چیست؟

هر Event داخل Partition یک شماره ترتیبی به نام Offset دارد.

Partition Offset 

Partition 0

Offset 0 → PaymentStarted

Offset 1 → PaymentCompleted

Offset 2 → PaymentRefunded

Consumer با نگهداری Offset مشخص می‌کند تا کدام Event را پردازش کرده است.

📌
Kafka Event را پس از خوانده شدن حذف نمی‌کند. Consumer فقط Offset خود را جلو می‌برد.

Consumer Group

چند Consumer می‌توانند با یک GroupId مشترک عضو یک Consumer Group شوند.

Kafka هر Partition را در یک Consumer Group فقط به یک Consumer اختصاص می‌دهد.

Consumer Group 

Topic: payment-completed

Partition 0 ───► Consumer 1

Partition 1 ───► Consumer 2

Partition 2 ───► Consumer 3

اگر تعداد Consumerها بیشتر از تعداد Partitionها باشد، بعضی Consumerها بدون Partition باقی می‌مانند.

⚠️
حداکثر میزان پردازش موازی در یک Consumer Group به تعداد Partitionهای Topic وابسته است.

Consumer در .NET

C# Consumer 
using Confluent.Kafka;

var config = new ConsumerConfig
{
    BootstrapServers = "localhost:9092",
    GroupId = "order-service-group",
    AutoOffsetReset = AutoOffsetReset.Earliest,
    EnableAutoCommit = false
};

using var consumer =
    new ConsumerBuilder<string, string>(config).Build();

consumer.Subscribe("payment-completed");

using var cancellationTokenSource = new CancellationTokenSource();

Console.CancelKeyPress += (_, eventArgs) =>
{
    eventArgs.Cancel = true;
    cancellationTokenSource.Cancel();
};

try
{
    while (!cancellationTokenSource.IsCancellationRequested)
    {
        var result = consumer.Consume(
            cancellationTokenSource.Token
        );

        Console.WriteLine(
            $"Partition: {result.Partition}, " +
            $"Offset: {result.Offset}"
        );

        Console.WriteLine(result.Message.Value);

        // پردازش موفق Event
        consumer.Commit(result);
    }
}
catch (OperationCanceledException)
{
    consumer.Close();
}

در این نمونه، Auto Commit غیرفعال شده و Offset پس از پردازش موفق Event ثبت می‌شود.

⚠️
اگر Offset قبل از تکمیل عملیات ثبت شود، ممکن است در صورت Crash شدن Consumer، Event پردازش‌نشده از دست برود.

Replication

Kafka برای افزایش تحمل خطا، از هر Partition چند نسخه روی Brokerهای مختلف نگهداری می‌کند.

Partition Replication 

Partition 0

   ├── Leader   → Broker 1
   ├── Replica  → Broker 2
   └── Replica  → Broker 3

عملیات خواندن و نوشتن روی Leader انجام می‌شود و Replicaها اطلاعات را از آن کپی می‌کنند.

اگر Broker مربوط به Leader از دسترس خارج شود، یکی از Replicaها می‌تواند Leader جدید شود.

✅
Replication باعث می‌شود خرابی یک Broker لزوماً باعث از دست رفتن Eventها نشود.

Kafka Controller

Kafka برای مدیریت Metadataهای Cluster، انتخاب Leader و تشخیص Brokerهای فعال به یک Controller نیاز دارد.

در معماری‌های جدید Kafka این هماهنگی با KRaft انجام می‌شود.

Cluster Coordination 

Kafka Controllers

      │

      ├── Manage Cluster Metadata
      ├── Detect Broker Failure
      └── Select Partition Leaders

جریان کامل یک Event

Complete Flow 

1. Producer creates an Event

2. Message Key selects a Partition

3. Producer sends Event to Partition Leader

4. Event is replicated to other Brokers

5. Consumer Group reads the Event

6. One Consumer processes the Partition

7. Consumer commits the Offset

نکات مهم طراحی

موضوع نکته
تعداد Partition میزان Parallelism و توسعه آینده را مشخص می‌کند.
Message Key روی Ordering و توزیع Eventها اثر می‌گذارد.
Replication Factor تحمل خطای Topic را مشخص می‌کند.
Offset Commit زمان ثبت آن باید با موفقیت پردازش هماهنگ باشد.
Consumer Group برای توزیع پردازش میان چند Consumer استفاده می‌شود.

اشتباهات رایج

❌
استفاده از یک Key ثابت برای تمام Eventها باعث می‌شود همه Messageها وارد یک Partition شوند و قابلیت پردازش موازی از بین برود.
❌
افزایش تعداد Consumerها بدون افزایش تعداد Partitionها لزوماً Performance را بیشتر نمی‌کند.
❌
Commit کردن Offset قبل از پردازش کامل Event ممکن است باعث از دست رفتن منطقی Message شود.

سؤالات مصاحبه

🎓
سؤال ۱: Broker در Kafka چیست؟

پاسخ: هر سرور Kafka یک Broker است که مسئول دریافت، ذخیره و ارائه Eventهای Topicها به Producer و Consumer است.
🎓
سؤال ۲: چرا Topic به چند Partition تقسیم می‌شود؟

پاسخ: برای توزیع داده میان Brokerها و فراهم کردن پردازش موازی، مقیاس‌پذیری و Throughput بیشتر.
🎓
سؤال ۳: آیا Kafka ترتیب تمام Eventهای یک Topic را تضمین می‌کند؟

پاسخ: خیر. ترتیب فقط داخل هر Partition تضمین می‌شود. برای حفظ ترتیب Eventهای یک Entity باید از Key یکسان استفاده شود.
🎓
سؤال ۴: اگر تعداد Consumerها بیشتر از Partitionها باشد چه اتفاقی می‌افتد؟

پاسخ: در یک Consumer Group، هر Partition فقط به یک Consumer اختصاص داده می‌شود؛ بنابراین Consumerهای اضافه بیکار باقی می‌مانند.
🎓
سؤال ۵: Offset چه کاربردی دارد؟

پاسخ: Offset موقعیت Event در Partition را مشخص می‌کند و Consumer با ثبت آن می‌داند تا کدام Event را پردازش کرده است.
🎓
سؤال ۶: Replication چه مشکلی را حل می‌کند؟

پاسخ: با نگهداری نسخه‌های یک Partition روی Brokerهای مختلف، از دسترس‌پذیری سیستم در زمان خرابی Broker افزایش پیدا می‌کند.

قانون طلایی

🎯
در Kafka، انتخاب صحیح Partition، Message Key و Consumer Group مستقیماً روی Ordering، Performance و مقیاس‌پذیری سیستم اثر می‌گذارد.

جمع‌بندی

معماری Kafka از Broker، Topic، Partition، Producer، Consumer و Consumer Group تشکیل شده است. Eventها داخل Partitionها ذخیره می‌شوند و هر Event یک Offset مشخص دارد.

Partitionها امکان پردازش موازی را فراهم می‌کنند و Replication باعث افزایش تحمل خطا می‌شود. طراحی صحیح Message Key، تعداد Partitionها و نحوه Commit کردن Offset از مهم‌ترین بخش‌های کار با Kafka است.

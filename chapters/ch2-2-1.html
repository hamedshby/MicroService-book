سوالات مصاحبه gRPC - بخش ۱ (مفاهیم پایه)

❓ سوال ۱: gRPC چیست؟

gRPC یک فریمورک متن‌باز برای پیاده‌سازی Remote Procedure Call (RPC) است که توسط Google توسعه داده شده است. این فریمورک از HTTP/2 و Protocol Buffers برای برقراری ارتباط سریع، کم‌حجم و Type-Safe بین سرویس‌ها استفاده می‌کند.

❓ سوال ۲: gRPC مخفف چیست؟

gRPC مخفف Google Remote Procedure Call است. امروزه gRPC یک پروژه متن‌باز مستقل است و تنها به محصولات Google محدود نمی‌شود.

❓ سوال ۳: چرا Google پروژه gRPC را توسعه داد؟

Google برای کاهش Latency، کاهش حجم پیام‌ها، افزایش Performance و ایجاد ارتباط سریع بین سرویس‌های داخلی خود، gRPC را توسعه داد. استفاده از HTTP/2 و Protocol Buffers باعث شد gRPC نسبت به REST در ارتباط Service-to-Service کارایی بیشتری داشته باشد.

❓ سوال ۴: gRPC چگونه کار می‌کند؟

ابتدا قرارداد ارتباطی در فایل .proto تعریف می‌شود. سپس Client و Server از روی آن کلاس‌های موردنیاز را تولید می‌کنند. هنگام فراخوانی متد، داده‌ها با Protocol Buffers به صورت Binary Serialize شده، از طریق HTTP/2 ارسال می‌شوند و پاسخ مجدداً Deserialize شده و به صورت یک شیء در اختیار برنامه قرار می‌گیرد.

❓ سوال ۵: RPC چیست؟

RPC یا Remote Procedure Call روشی برای فراخوانی متدهای یک سرویس روی یک سیستم دیگر است، به گونه‌ای که برای برنامه‌نویس مانند فراخوانی یک متد محلی به نظر برسد. gRPC یکی از محبوب‌ترین پیاده‌سازی‌های RPC است.

❓ سوال ۶: تفاوت RPC و REST چیست؟

در REST ارتباط بر اساس Resource و Endpoint انجام می‌شود و معمولاً داده‌ها با JSON منتقل می‌شوند. اما در RPC ارتباط بر اساس فراخوانی متدها است و در gRPC داده‌ها توسط Protocol Buffers و از طریق HTTP/2 منتقل می‌شوند. به همین دلیل gRPC معمولاً سرعت و کارایی بیشتری نسبت به REST دارد.

❓ سوال ۷: چه زمانی gRPC را انتخاب می‌کنید؟

زمانی که ارتباط بین Microserviceها نیاز به Performance بالا، Latency پایین، حجم کم پیام‌ها، Type Safety و Streaming داشته باشد، gRPC بهترین انتخاب است. این فناوری بیشتر برای ارتباط Service-to-Service در شبکه داخلی استفاده می‌شود.

❓ سوال ۸: چه زمانی REST را انتخاب می‌کنید؟

REST برای APIهای عمومی، ارتباط با Browser، Mobile Application، سیستم‌های Third-Party و سرویس‌هایی که نیاز به سازگاری بالا و خوانایی پیام‌ها دارند، انتخاب مناسب‌تری است.

❓ سوال ۹: مزایای gRPC چیست؟

از مهم‌ترین مزایای gRPC می‌توان به Performance بالا، استفاده از HTTP/2، حجم کم پیام‌ها، Protocol Buffers، پشتیبانی از Streaming، تولید خودکار Client و Server، Type Safety، ارتباط سریع بین Microserviceها و مصرف کمتر CPU و پهنای باند اشاره کرد.

❓ سوال ۱۰: معایب gRPC چیست؟

gRPC برای Browserها پشتیبانی محدودی دارد، پیام‌های Binary آن مانند JSON قابل خواندن نیستند، نیاز به فایل Proto دارد و برای APIهای عمومی معمولاً به اندازه REST مناسب نیست. همچنین تست دستی آن نسبت به REST پیچیده‌تر است.

❓ سوال ۱۱: Protocol Buffers چیست؟

Protocol Buffers یا Protobuf یک مکانیزم Serialize و Deserialize داده‌ها است که توسط Google توسعه داده شده است. این فناوری داده‌ها را به صورت Binary ذخیره و منتقل می‌کند، بنابراین نسبت به JSON حجم کمتری داشته و سرعت پردازش بالاتری دارد.

❓ سوال ۱۲: Proto File چیست؟

فایل .proto قرارداد (Contract) بین Client و Server را مشخص می‌کند. در این فایل Messageها، Serviceها، Enumها و RPC Methodها تعریف می‌شوند و بر اساس آن کلاس‌های موردنیاز در زبان‌های مختلف به صورت خودکار تولید می‌شوند.

❓ سوال ۱۳: چرا gRPC از Protocol Buffers استفاده می‌کند؟

زیرا Protocol Buffers نسبت به JSON حجم کمتری دارد، سریع‌تر Serialize و Deserialize می‌شود، مصرف CPU و پهنای باند را کاهش می‌دهد و ارتباط بین سرویس‌ها را سریع‌تر و بهینه‌تر می‌کند.

❓ سوال ۱۴: Code Generation در gRPC چگونه انجام می‌شود؟

هنگام Build پروژه، فایل .proto توسط ابزار Grpc.Tools پردازش می‌شود و کلاس‌های C# شامل Messageها، Client، Service Base و سایر کلاس‌های موردنیاز به صورت خودکار تولید می‌شوند. به همین دلیل نیازی به نوشتن دستی DTOها و Clientها نیست.

❓ سوال ۱۵: Contract First چیست؟

Contract First روشی در طراحی API است که ابتدا قرارداد ارتباطی در فایل .proto تعریف می‌شود و سپس Client و Server بر اساس همان قرارداد پیاده‌سازی می‌شوند. این روش باعث یکپارچگی، جلوگیری از ناسازگاری بین سرویس‌ها و ساده‌تر شدن توسعه سیستم‌های توزیع‌شده می‌شود.

❓ سوال ۱۶: HTTP/2 چیست؟

HTTP/2 نسخه دوم پروتکل HTTP است که نسبت به HTTP/1.1 امکاناتی مانند Multiplexing، Header Compression، Binary Protocol و Server Push را ارائه می‌دهد. gRPC به صورت پیش‌فرض روی HTTP/2 اجرا می‌شود تا ارتباطی سریع‌تر و با تأخیر کمتر بین سرویس‌ها برقرار کند.

❓ سوال ۱۷: تفاوت HTTP/1.1 و HTTP/2 چیست؟

در HTTP/1.1 معمولاً هر Connection تنها یک درخواست را در یک زمان پردازش می‌کند، اما HTTP/2 امکان ارسال همزمان چندین درخواست روی یک Connection را فراهم می‌کند. همچنین HTTP/2 از Binary Protocol و Header Compression استفاده می‌کند که باعث کاهش مصرف پهنای باند و افزایش Performance می‌شود.

❓ سوال ۱۸: Multiplexing چیست؟

Multiplexing قابلیتی در HTTP/2 است که اجازه می‌دهد چندین Request و Response به صورت همزمان و مستقل روی یک Connection منتقل شوند. این ویژگی باعث کاهش تعداد Connectionهای موردنیاز و افزایش سرعت ارتباط بین سرویس‌ها می‌شود.

❓ سوال ۱۹: چرا Multiplexing سرعت gRPC را افزایش می‌دهد؟

زیرا دیگر نیازی به ایجاد Connection جدید برای هر درخواست نیست. چندین درخواست می‌توانند به صورت همزمان روی یک Connection ارسال شوند، بنابراین زمان انتظار، سربار ایجاد Connection و Latency کاهش پیدا می‌کند و Throughput سیستم افزایش می‌یابد.

❓ سوال ۲۰: Binary Protocol چیست؟

Binary Protocol روشی برای انتقال داده‌ها به صورت Binary است، نه متن (Text). gRPC از Protocol Buffers برای تولید داده‌های Binary استفاده می‌کند که حجم کمتری نسبت به JSON دارند و سریع‌تر Serialize و Deserialize می‌شوند.

💡
در بخش بعد با تفاوت Binary و JSON، Header Compression، الگوریتم HPACK، دلیل استفاده gRPC از HTTP/2 و امکان اجرای gRPC روی HTTP/1.1 آشنا خواهید شد.

سوالات مصاحبه gRPC - بخش ۵ (HTTP/2 و Performance)

❓ سوال ۲۱: تفاوت Binary و JSON چیست؟

JSON یک فرمت متنی (Text-Based) است که خوانایی بالایی برای انسان دارد، اما حجم بیشتری تولید می‌کند. در مقابل، Binary یک فرمت دودویی است که حجم کمتری دارد، سریع‌تر Serialize و Deserialize می‌شود و مصرف CPU و پهنای باند را کاهش می‌دهد. به همین دلیل gRPC از Protocol Buffers و Binary Protocol استفاده می‌کند.

❓ سوال ۲۲: Header Compression چیست؟

Header Compression قابلیتی در HTTP/2 است که هدرهای تکراری HTTP را فشرده می‌کند. این کار باعث کاهش حجم داده‌های ارسالی، مصرف کمتر پهنای باند و افزایش سرعت ارتباط، به‌ویژه در سیستم‌هایی با تعداد زیاد درخواست، می‌شود.

❓ سوال ۲۳: HPACK چیست؟

HPACK الگوریتم فشرده‌سازی Headerها در HTTP/2 است. این الگوریتم با نگهداری مقادیر تکراری در یک جدول (Header Table)، از ارسال مجدد هدرهای یکسان جلوگیری می‌کند و باعث کاهش حجم درخواست‌ها و پاسخ‌ها می‌شود.

❓ سوال ۲۴: چرا gRPC از HTTP/2 استفاده می‌کند؟

زیرا HTTP/2 قابلیت‌هایی مانند Multiplexing، Binary Protocol، Header Compression و Streaming را به صورت بومی فراهم می‌کند. این ویژگی‌ها باعث کاهش Latency، افزایش Throughput و بهبود Performance در ارتباط بین Microserviceها می‌شوند.

❓ سوال ۲۵: آیا gRPC روی HTTP/1.1 کار می‌کند؟

gRPC استاندارد برای عملکرد کامل خود به HTTP/2 نیاز دارد و روی HTTP/1.1 اجرا نمی‌شود. البته برای مرورگرها می‌توان از gRPC-Web استفاده کرد که درخواست‌ها را از طریق Proxy به gRPC استاندارد تبدیل می‌کند.

💡
در بخش بعد با سؤالات مربوط به Latency، کاهش مصرف CPU، حجم پیام‌ها، مقایسه Performance با REST و زمان‌هایی که استفاده از gRPC بیشترین تأثیر را دارد آشنا خواهید شد.

سوالات مصاحبه gRPC - بخش ۶ (HTTP/2 و Performance)

❓ سوال ۲۶: Latency در gRPC چگونه کاهش پیدا می‌کند؟

gRPC با استفاده از HTTP/2، قابلیت Multiplexing، Protocol Buffers و Header Compression میزان تأخیر (Latency) را کاهش می‌دهد. همچنین به دلیل استفاده از یک Connection مشترک، سربار ایجاد Connectionهای متعدد نیز حذف می‌شود.

❓ سوال ۲۷: چرا حجم پیام‌های gRPC کمتر است؟

زیرا gRPC به جای JSON از Protocol Buffers استفاده می‌کند. Protobuf داده‌ها را به صورت Binary ذخیره و منتقل می‌کند که نسبت به JSON حجم بسیار کمتری دارد و در نتیجه مصرف پهنای باند کاهش می‌یابد.

❓ سوال ۲۸: چرا gRPC مصرف CPU کمتری دارد؟

Protocol Buffers نسبت به JSON عملیات Serialize و Deserialize سریع‌تری دارد و پردازش کمتری روی CPU انجام می‌دهد. همچنین Binary بودن پیام‌ها باعث کاهش زمان پردازش و افزایش کارایی سیستم می‌شود.

❓ سوال ۲۹: آیا همیشه gRPC سریع‌تر از REST است؟

خیر. gRPC معمولاً در ارتباط‌های داخلی بین Microserviceها و زمانی که تعداد درخواست‌ها زیاد باشد، عملکرد بهتری دارد. اما برای APIهای عمومی، Browserها یا سرویس‌های خارجی، REST به دلیل سازگاری بیشتر و پشتیبانی گسترده‌تر می‌تواند انتخاب مناسب‌تری باشد.

❓ سوال ۳۰: چه زمانی افزایش Performance gRPC محسوس است؟

زمانی که تعداد درخواست‌ها زیاد باشد، حجم داده‌ها بالا باشد یا ارتباط مداوم بین Microserviceها برقرار شود، مزایای gRPC کاملاً محسوس خواهد بود. در این شرایط استفاده از HTTP/2 و Protocol Buffers باعث کاهش Latency، افزایش Throughput و مصرف کمتر منابع نسبت به REST می‌شود.

🎯
با پایان این بخش، مفاهیم مربوط به HTTP/2، Performance، Latency، Binary Protocol، Header Compression و تفاوت‌های عملکردی gRPC و REST را مرور کردید. در بخش بعد وارد سؤالات مربوط به Proto File، Message، Service، Enum و Code Generation خواهیم شد.

سوالات مصاحبه gRPC - بخش ۷ (Proto File)

❓ سوال ۳۱: Proto File چیست؟

Proto File فایلی با پسوند .proto است که قرارداد (Contract) بین Client و Server را تعریف می‌کند. در این فایل Messageها، Serviceها، Enumها و RPC Methodها مشخص می‌شوند و بر اساس آن کلاس‌های موردنیاز به صورت خودکار تولید می‌شوند.

❓ سوال ۳۲: syntax = "proto3" یعنی چه؟

این عبارت نسخه سینتکس فایل Proto را مشخص می‌کند. امروزه تقریباً تمام پروژه‌های gRPC از proto3 استفاده می‌کنند، زیرا نسخه استاندارد، ساده‌تر و سازگارتر با زبان‌های مختلف برنامه‌نویسی است.

❓ سوال ۳۳: package چیست؟

کلمه package فضای نام (Namespace) فایل Proto را مشخص می‌کند و از تداخل نام Serviceها و Messageها جلوگیری می‌کند. این مقدار در تولید کلاس‌ها و مدیریت قراردادها نیز کاربرد دارد.

❓ سوال ۳۴: csharp_namespace چیست؟

گزینه csharp_namespace تعیین می‌کند کلاس‌های C# تولید شده در چه Namespaceای قرار بگیرند. این گزینه باعث می‌شود ساختار پروژه منظم‌تر باشد و Namespace کلاس‌های تولید شده با ساختار پروژه هماهنگ شود.

❓ سوال ۳۵: Message چیست؟

Message ساختاری برای تعریف داده‌های ورودی و خروجی در gRPC است و معادل DTO یا Model در ASP.NET Core محسوب می‌شود. هر Message از تعدادی Field تشکیل شده که هر کدام دارای یک شماره یکتا هستند.

💡
در بخش بعد با Enum، Service، RPC Method و اهمیت شماره Fieldها آشنا خواهید شد.

سوالات مصاحبه gRPC - بخش ۸ (Proto File)

❓ سوال ۳۶: Enum چیست؟

Enum مجموعه‌ای از مقادیر ثابت و از پیش تعریف‌شده است که برای نمایش وضعیت‌ها یا مقادیر محدود استفاده می‌شود. استفاده از Enum باعث خوانایی بیشتر کد، جلوگیری از استفاده از اعداد جادویی (Magic Numbers) و افزایش Type Safety می‌شود.

❓ سوال ۳۷: Service چیست؟

Service مجموعه‌ای از RPC Methodها را تعریف می‌کند و مشخص می‌کند Client چه متدهایی را می‌تواند فراخوانی کند. از نظر مفهومی، Service در gRPC مشابه Controller در REST است.

❓ سوال ۳۸: RPC Method چیست؟

RPC Method متدی است که در داخل یک Service تعریف می‌شود و توسط Client قابل فراخوانی است. هر RPC دارای نوع ورودی (Request) و خروجی (Response) مشخصی است و می‌تواند از نوع Unary یا Streaming باشد.

❓ سوال ۳۹: چرا شماره Fieldها مهم هستند؟

در Protocol Buffers هر Field دارای یک شماره یکتا است که در فرآیند Serialize و Deserialize استفاده می‌شود. این شماره‌ها بخشی از قرارداد Binary هستند و برای سازگاری بین نسخه‌های مختلف سرویس اهمیت زیادی دارند.

❓ سوال ۴۰: آیا می‌توان شماره Field را تغییر داد؟

خیر. پس از انتشار API نباید شماره Fieldها را تغییر داد، زیرا این کار باعث ناسازگاری بین Client و Server می‌شود. اگر نیاز به افزودن فیلد جدید باشد، باید از یک شماره جدید استفاده کرد و شماره‌های قبلی را حفظ نمود.

⚠️
یکی از مهم‌ترین قوانین طراحی Proto این است که پس از انتشار سرویس، شماره Fieldها را تغییر ندهید و از استفاده مجدد شماره‌های حذف‌شده خودداری کنید.

سوالات مصاحبه gRPC - بخش ۹ (Proto File)

❓ سوال ۴۱: چگونه Property جدید به Message اضافه کنیم؟

برای اضافه کردن Property جدید، یک Field با شماره (Field Number) جدید به انتهای Message اضافه می‌کنیم. این کار با نسخه‌های قبلی سازگار است و باعث ایجاد Breaking Change نمی‌شود.

❓ سوال ۴۲: Breaking Change چیست؟

Breaking Change تغییری است که باعث ناسازگاری بین Client و Server می‌شود. تغییر شماره Fieldها، حذف Fieldهای مورد استفاده، تغییر نوع داده (Data Type) یا تغییر نام Service و RPC Method از جمله نمونه‌های Breaking Change در gRPC هستند.

❓ سوال ۴۳: Non-Breaking Change چیست؟

Non-Breaking Change تغییری است که بدون ایجاد مشکل برای Clientهای قدیمی قابل اعمال باشد. اضافه کردن Field جدید با شماره جدید، اضافه کردن RPC Method جدید یا افزودن مقدار جدید به Enum از نمونه‌های Non-Breaking Change محسوب می‌شوند.

❓ سوال ۴۴: چگونه Versioning در gRPC انجام می‌شود؟

Versioning معمولاً با ایجاد فایل Proto جدید، استفاده از Packageهای جداگانه (مانند v1 و v2) یا تعریف Serviceهای نسخه‌بندی‌شده انجام می‌شود. تا حد امکان باید از تغییرات سازگار (Backward Compatible) استفاده کرد و از ایجاد Breaking Change جلوگیری نمود.

❓ سوال ۴۵: بهترین روش طراحی Proto چیست؟

از proto3 استفاده کنید، شماره Fieldها را تغییر ندهید، برای هر Message مسئولیت مشخصی در نظر بگیرید، از Enum برای وضعیت‌ها استفاده کنید، فایل‌های Proto را کوچک و ماژولار نگه دارید و Contract را به گونه‌ای طراحی کنید که در آینده بدون ایجاد Breaking Change قابل توسعه باشد.

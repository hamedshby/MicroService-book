🌐 ارتباط همزمان (Synchronous)

2.1 HTTP REST

📅 به‌روزرسانی: ۱۴۰۵/۰۴/۱۶
⏱️ زمان مطالعه: ۲۰ دقیقه
👤 تیم فنی

بعد از آشنایی با معماری Microservices و چالش‌های سیستم‌های توزیع‌شده، اولین روش ارتباط بین سرویس‌ها که باید یاد بگیریم، HTTP REST است.

امروزه REST پرکاربردترین روش ارتباط بین Clientها و APIها و همچنین یکی از رایج‌ترین روش‌های ارتباط همزمان (Synchronous Communication) بین Microserviceها است.

تقریباً تمام برنامه‌هایی که روزانه از آن‌ها استفاده می‌کنیم، از جمله فروشگاه‌های اینترنتی، بانکداری آنلاین، اپلیکیشن‌های موبایل و سیستم‌های سازمانی، حداقل بخشی از ارتباطات خود را از طریق REST انجام می‌دهند.

💡
REST ساده‌ترین و رایج‌ترین روش ارتباط بین سرویس‌ها است، اما همیشه بهترین انتخاب نیست. در فصل‌های بعد خواهیم دید که در برخی سناریوها gRPC یا Message Broker گزینه مناسب‌تری هستند.

REST چیست؟

REST مخفف عبارت Representational State Transfer است.

REST یک سبک معماری (Architectural Style) برای طراحی API است، نه یک پروتکل و نه یک کتابخانه.

در REST معمولاً ارتباط از طریق پروتکل HTTP انجام می‌شود و منابع سیستم (Resources) با استفاده از URLها در اختیار Client قرار می‌گیرند.

REST Example 
GET /api/products

GET /api/orders/105

POST /api/orders

PUT /api/products/12

DELETE /api/products/12
✅
در REST هر URL معمولاً نماینده یک Resource است؛ مانند Product، Order یا Customer.

Request / Response

REST بر پایه الگوی Request / Response کار می‌کند.

ابتدا Client یک درخواست (Request) برای Server ارسال می‌کند و سپس تا دریافت پاسخ (Response) منتظر می‌ماند.

Request / Response

به همین دلیل REST یک روش ارتباط Synchronous محسوب می‌شود؛ زیرا Client بدون دریافت پاسخ قادر به ادامه عملیات نیست.

⚠️
اگر Server پاسخ ندهد یا Timeout رخ دهد، Client نیز متوقف خواهد شد.

ساختار HTTP Request

هر درخواست HTTP از چند بخش اصلی تشکیل شده است.

HTTP Request 
POST /api/orders HTTP/1.1

Host: api.shop.com

Authorization: Bearer eyJhb...

Content-Type: application/json

Accept: application/json

{
    "customerId":15,
    "productId":22,
    "count":2
}

بخش‌های اصلی Request

بخش توضیح
Method نوع عملیات مانند GET یا POST
URL آدرس Resource
Headers اطلاعات اضافی مانند Token و Content-Type
Body داده ارسالی (معمولاً JSON)
📌
در درخواست‌های GET معمولاً Body وجود ندارد، اما در POST و PUT اغلب اطلاعات در Body ارسال می‌شوند.

ساختار HTTP Response

پس از پردازش درخواست، Server یک Response به Client ارسال می‌کند.

HTTP Response

Content-Type: application/json

Content-Length: 145

{
    "id":105,
    "status":"Success",
    "message":"Order Created"
}

مزایای REST

  • سادگی در طراحی و پیاده‌سازی
  • پشتیبانی توسط تمام زبان‌های برنامه‌نویسی
  • مستقل از Platform
  • قابلیت تست آسان با Postman و Swagger
  • استفاده از استاندارد HTTP
  • پشتیبانی مناسب توسط مرورگرها
  • مستندات گسترده و جامعه کاربری بزرگ
  • مناسب برای ارتباط Client با Server
✅
اگر سرعت بسیار بالا یا Streaming مورد نیاز نباشد، REST معمولاً اولین انتخاب برای طراحی APIهای عمومی است.

معایب REST

  • Overhead بیشتر نسبت به gRPC
  • ارسال داده به صورت متنی (JSON)
  • Latency بالاتر نسبت به ارتباط‌های Binary
  • عدم پشتیبانی ذاتی از Streaming دوطرفه
  • مناسب نبودن برای ارتباط‌های بسیار پرتعداد بین سرویس‌ها
  • وابستگی به HTTP Request/Response
  • نیاز به برقراری ارتباط شبکه برای هر درخواست
⚠️
REST انتخاب بسیار خوبی برای APIهای عمومی و ارتباط با Clientها است، اما برای ارتباط‌های پرتعداد بین Microserviceها معمولاً gRPC یا Message Brokerها کارایی و بهره‌وری بیشتری دارند.
🎯
در بخش بعد با متدهای HTTP (GET، POST، PUT، PATCH و DELETE)، مفهوم Safe و Idempotent Methods و همچنین Status Codeهای مهم HTTP آشنا خواهیم شد.

HTTP Methods

در REST، هر درخواست HTTP با یک HTTP Method آغاز می‌شود. این Method مشخص می‌کند که Client قصد انجام چه عملیاتی روی یک Resource را دارد.

به عنوان مثال، ممکن است بخواهیم اطلاعات یک محصول را دریافت کنیم، سفارش جدیدی ثبت کنیم یا اطلاعات یک مشتری را ویرایش کنیم.

HTTP Methods 
GET     /api/products

POST    /api/orders

PUT     /api/products/15

PATCH   /api/users/8

DELETE  /api/products/15
💡
انتخاب صحیح HTTP Method باعث خوانایی بهتر API و رعایت استاندارد REST خواهد شد.

GET

متد GET برای دریافت اطلاعات استفاده می‌شود و نباید باعث تغییر داده‌های سیستم شود.

Example 
GET /api/products

GET /api/orders/105
✅
GET فقط داده را برمی‌گرداند و نباید عملیات Create، Update یا Delete انجام دهد.

POST

متد POST برای ایجاد Resource جدید یا اجرای عملیاتی استفاده می‌شود که باعث تغییر وضعیت سیستم خواهد شد.

Example 
POST /api/orders

{
    "customerId":15,
    "productId":20,
    "count":2
}

هر بار ارسال POST ممکن است نتیجه متفاوتی ایجاد کند.

PUT

متد PUT برای جایگزین کردن کامل یک Resource استفاده می‌شود.

Example 
PUT /api/products/15

{
    "name":"Laptop",
    "price":950,
    "stock":8
}

در PUT معمولاً تمام اطلاعات Resource ارسال می‌شود.

PATCH

اگر فقط بخشی از اطلاعات نیاز به تغییر داشته باشد، از PATCH استفاده می‌کنیم.

Example 
PATCH /api/products/15

{
    "price":1100
}

PATCH تنها فیلدهای ارسال‌شده را تغییر می‌دهد.

DELETE

متد DELETE برای حذف یک Resource استفاده می‌شود.

Example 
DELETE /api/products/15

Safe Methods

متدهای Safe هیچ تغییری در وضعیت سیستم ایجاد نمی‌کنند.

Method Safe؟
GET ✔
HEAD ✔
OPTIONS ✔
POST ✖
PUT ✖
PATCH ✖
DELETE ✖
📌
Safe بودن به معنی عدم تغییر وضعیت سیستم است، نه اینکه عملیات سریع یا بدون هزینه باشد.

Idempotent Methods

یک متد زمانی Idempotent است که اجرای چندباره آن، نتیجه‌ای مشابه اجرای یک‌باره داشته باشد.

Method Idempotent؟
GET ✔
PUT ✔
DELETE ✔
HEAD ✔
OPTIONS ✔
PATCH بستگی به پیاده‌سازی دارد
POST ✖

مثال

PUT 
PUT /api/products/10

{
    "price":500
}

اگر این درخواست ده بار ارسال شود، قیمت همچنان ۵۰۰ خواهد بود.

POST 
POST /api/orders

اگر این درخواست ده بار ارسال شود، احتمالاً ده سفارش جدید ایجاد خواهد شد.

⚠️
در طراحی REST API، استفاده از Method مناسب نقش مهمی در جلوگیری از خطاهای تکراری و Retryهای ناخواسته دارد.

HTTP Status Codes

Status Code نتیجه پردازش درخواست را مشخص می‌کند.

کد نام توضیح
200 OK درخواست با موفقیت انجام شد.
201 Created Resource جدید ایجاد شد.
204 No Content عملیات موفق بود اما داده‌ای برنمی‌گردد.
400 Bad Request درخواست نامعتبر است.
401 Unauthorized کاربر احراز هویت نشده است.
403 Forbidden کاربر مجوز دسترسی ندارد.
404 Not Found Resource پیدا نشد.
409 Conflict تداخل با وضعیت فعلی سیستم.
422 Unprocessable Entity داده معتبر است اما از نظر قوانین کسب‌وکار قابل پردازش نیست.
500 Internal Server Error خطای داخلی سرور.
503 Service Unavailable سرویس موقتاً در دسترس نیست.

Best Practices

  • برای دریافت اطلاعات همیشه از GET استفاده کنید.
  • برای ایجاد Resource جدید از POST استفاده کنید.
  • برای جایگزینی کامل Resource از PUT استفاده کنید.
  • برای تغییر بخشی از Resource از PATCH استفاده کنید.
  • برای حذف Resource از DELETE استفاده کنید.
  • از Status Code مناسب استفاده کنید و همیشه 200 برنگردانید.
  • پیام‌های خطا را با ساختار یکسان (مانند ProblemDetails در ASP.NET Core) برگردانید.
  • APIها را تا حد امکان Stateless طراحی کنید.
  • از URLهای مبتنی بر Resource استفاده کنید، نه نام عملیات (مانند /products به‌جای /getProducts).
  • برای عملیات‌های حساس که ممکن است Retry شوند، مکانیزم Idempotency را در نظر بگیرید.
🎯
انتخاب صحیح HTTP Method و Status Code باعث می‌شود API قابل پیش‌بینی‌تر، استانداردتر و سازگارتر با ابزارها و Frameworkهای مختلف باشد. این موضوع در طراحی Microserviceها اهمیت زیادی دارد، زیرا سرویس‌های دیگر نیز بر اساس همین قراردادها با API شما تعامل خواهند کرد.

HttpClient

در پروژه‌های ASP.NET Core، متداول‌ترین راه برای ارسال درخواست HTTP به سایر APIها، استفاده از کلاس HttpClient است.

تقریباً تمام ارتباط‌های REST بین Microserviceها در .NET توسط HttpClient انجام می‌شوند.

Simple Example 
var client = new HttpClient();

var response = await client.GetAsync(
    "https://api.payment.com/api/payments/10");

response.EnsureSuccessStatusCode();

var json = await response.Content.ReadAsStringAsync();

در نگاه اول استفاده از HttpClient بسیار ساده به نظر می‌رسد، اما استفاده اشتباه از آن می‌تواند باعث مشکلات جدی در سیستم‌های Production شود.

⚠️
یکی از رایج‌ترین اشتباهات برنامه‌نویسان .NET، ساختن HttpClient جدید برای هر درخواست است.

چرخه عمر HttpClient

هر HttpClient دارای Connection Pool مخصوص به خود است.

وقتی یک HttpClient ساخته می‌شود، ارتباط‌های TCP را مدیریت می‌کند و در صورت امکان آن‌ها را مجدداً استفاده (Reuse) می‌کند.

Payment API

اگر برای هر درخواست یک HttpClient جدید ساخته شود، Connection Pool نیز دوباره ساخته خواهد شد.

استفاده اشتباه از HttpClient

کد زیر یکی از رایج‌ترین اشتباهات در پروژه‌های .NET است.

Bad Practice 
public async Task<string> GetProducts()
{
    using var client = new HttpClient();

    return await client.GetStringAsync(
        "https://api.shop.com/products");
}

در این مثال برای هر درخواست، یک HttpClient جدید ایجاد و بلافاصله Dispose می‌شود.

⚠️
Dispose شدن HttpClient به معنی بسته شدن فوری Connectionهای TCP نیست و همین موضوع می‌تواند باعث مشکلات شبکه شود.

Socket Exhaustion

یکی از معروف‌ترین مشکلات HttpClient، خطای Socket Exhaustion است.

هر بار که HttpClient جدید ساخته می‌شود، یک یا چند Socket جدید ایجاد می‌شود.

پس از پایان درخواست، این Socketها بلافاصله آزاد نمی‌شوند و برای مدتی در حالت TIME_WAIT باقی می‌مانند.

اگر سیستم هزاران درخواست ارسال کند، تمام Socketهای سیستم اشغال خواهند شد.

⚠️
Socket Exhaustion یکی از رایج‌ترین دلایل کاهش Performance و خطاهای ارتباطی در سیستم‌های پرترافیک است.

📌
این مشکل یکی از مهم‌ترین دلایل معرفی IHttpClientFactory در ASP.NET Core است.

نمونه صحیح استفاده (Singleton)

در نسخه‌های قدیمی .NET پیشنهاد می‌شد که HttpClient به صورت Singleton ایجاد شود.

Singleton 
public class PaymentClient
{
    private static readonly HttpClient client =
        new HttpClient();

    public async Task<string> GetAsync()
    {
        return await client.GetStringAsync(
            "https://api.payment.com");
    }
}

این روش مشکل Socket Exhaustion را تا حد زیادی برطرف می‌کند، اما همچنان مشکل به‌روزرسانی DNS را حل نمی‌کند.

نمونه استفاده در .NET

GET Request 
using var client = new HttpClient();

var response = await client.GetAsync(
    "https://api.shop.com/products");

if (response.IsSuccessStatusCode)
{
    var json =
        await response.Content.ReadAsStringAsync();

    Console.WriteLine(json);
}

ارسال درخواست POST

POST Request 
var client = new HttpClient();

var order = new
{
    CustomerId = 5,
    ProductId = 12,
    Count = 2
};

var response = await client.PostAsJsonAsync(
    "https://api.shop.com/orders",

response.EnsureSuccessStatusCode();

Bearer Token

مشکلات HttpClient به صورت خلاصه

مشکل توضیح

DNS Issue استفاده از IP قدیمی
Connection Leak مدیریت نادرست Connectionها
Latency عدم استفاده از Connection Pool
Performance ساخت مداوم HttpClient

Best Practices

  • برای هر درخواست HttpClient جدید ایجاد نکنید.
  • از IHttpClientFactory در ASP.NET Core استفاده کنید.
  • Timeout مناسب برای هر Client تنظیم کنید.
  • از CancellationToken برای لغو درخواست‌های غیرضروری استفاده کنید.
  • از EnsureSuccessStatusCode() یا بررسی Status Code استفاده کنید.
  • از Connection Pool استفاده کنید.
  • برای Retry از Polly یا IHttpClientFactory استفاده کنید، نه حلقه‌های دستی.
  • Authorization Header را فقط در صورت نیاز ارسال کنید.
  • برای جلوگیری از بن‌بست (Deadlock)، در کتابخانه‌ها از فراخوانی‌های همگام مانند .Result یا .Wait() روی متدهای async خودداری کنید.
  • در سرویس‌های Production، درخواست‌ها و پاسخ‌ها را با رعایت ملاحظات امنیتی لاگ کنید و از ثبت اطلاعات حساس مانند Token یا رمز عبور خودداری کنید.
🎯
اگرچه HttpClient کلاس اصلی ارسال درخواست HTTP در .NET است، اما در پروژه‌های ASP.NET Core امروزی، استفاده مستقیم از آن برای بیشتر سناریوها توصیه نمی‌شود. بهترین روش، استفاده از IHttpClientFactory است که مشکلات Socket Exhaustion، مدیریت Connection و به‌روزرسانی DNS را به شکل استاندارد حل می‌کند. در بخش بعد، IHttpClientFactory، Named Client و Typed Client را به‌صورت کامل بررسی خواهیم کرد.

IHttpClientFactory

از نسخه ASP.NET Core 2.1 به بعد، مایکروسافت کلاس IHttpClientFactory را برای مدیریت استاندارد HttpClient معرفی کرد.

هدف اصلی این Factory، حل مشکلاتی مانند Socket Exhaustion، DNS Refresh، مدیریت Connection Pool و یکپارچه‌سازی Policyهایی مانند Retry و Circuit Breaker است.

✅
در پروژه‌های ASP.NET Core، تقریباً همیشه استفاده از IHttpClientFactory به جای ساخت مستقیم HttpClient توصیه می‌شود.

ثبت IHttpClientFactory

Program.cs 
builder.Services.AddHttpClient();

استفاده از IHttpClientFactory

Example 
public class ProductService
{
    private readonly IHttpClientFactory _factory;

    public ProductService(IHttpClientFactory factory)
    {
        _factory = factory;
    }

    public async Task<string> GetProductsAsync()
    {
        var client = _factory.CreateClient();

        return await client.GetStringAsync(
            "https://api.shop.com/products");
    }
}

Named Client

اگر چند API مختلف داشته باشیم، می‌توانیم برای هر کدام یک Client با تنظیمات اختصاصی ایجاد کنیم.

Registration 
builder.Services.AddHttpClient(
    "Payment",
    client =>
    {
        client.BaseAddress =
            new Uri("https://payment-api.com");

        client.Timeout =
            TimeSpan.FromSeconds(10);
    });

استفاده از Named Client:

Usage 
var client =
    factory.CreateClient("Payment");

var response =
    await client.GetAsync("/api/payments");
💡
Named Client زمانی مناسب است که چند سرویس مختلف با تنظیمات متفاوت (مانند BaseAddress، Timeout یا Headerها) داشته باشید.

Typed Client

در پروژه‌های Enterprise معمولاً منطق ارتباط با هر سرویس در یک کلاس مجزا قرار می‌گیرد.

به این روش Typed Client گفته می‌شود.

Registration 
builder.Services.AddHttpClient<
    PaymentClient>(client =>
{
    client.BaseAddress =
        new Uri("https://payment-api.com");
});
Typed Client 
public class PaymentClient
{
    private readonly HttpClient _client;

    public PaymentClient(HttpClient client)
    {
        _client = client;
    }

    public async Task<PaymentDto> GetAsync(int id)
    {
        return await _client
            .GetFromJsonAsync<PaymentDto>(
                $"api/payments/{id}");
    }
}
✅
Typed Client بهترین گزینه برای پروژه‌های بزرگ است، زیرا تمام منطق ارتباط با یک API در یک کلاس متمرکز می‌شود.

Generated Client (Refit)

نوشتن کدهای تکراری برای فراخوانی APIها می‌تواند خسته‌کننده باشد.

کتابخانه Refit با استفاده از Interfaceها، Client مورد نیاز را به صورت خودکار تولید می‌کند.

Interface 
public interface IPaymentApi
{
    [Get("/api/payments/{id}")]
    Task<PaymentDto> GetAsync(int id);
}
ثبت سرویس:
Registration 
builder.Services
    .AddRefitClient<IPaymentApi>()
    .ConfigureHttpClient(c =>
    {
        c.BaseAddress =
            new Uri("https://payment-api.com");
    });
📌
Refit باعث کاهش Boilerplate Code می‌شود و نگهداری Clientها را ساده‌تر می‌کند.

Polly Integration

بررسی Retry Policy با Polly

یکی از قابلیت‌های مهم Polly، پیاده‌سازی Retry برای خطاهای موقتی (Transient Faults) است.

در مثال زیر، برای HttpClient یک Retry Policy تعریف شده است که در صورت بروز خطا، درخواست را حداکثر سه بار مجدداً ارسال می‌کند.

Retry Policy 
builder.Services
    .AddHttpClient("Payment")
    .AddTransientHttpErrorPolicy(policy =>
        policy.WaitAndRetryAsync(
            3,
            retryAttempt =>
                TimeSpan.FromSeconds(
                    Math.Pow(2, retryAttempt))));

Best Practice

  • Retry را فقط برای خطاهای موقتی انجام دهید.
  • از Exponential Backoff به جای Retry فوری استفاده کنید.
  • تعداد Retryها را محدود نگه دارید.
  • برای جلوگیری از Retry Storm از Jitter استفاده کنید.
  • Retry را با Circuit Breaker ترکیب کنید.
  • برای عملیات‌های حساس، Consumerها را Idempotent طراحی کنید.

Retry + Circuit Breaker

یکی از رایج‌ترین الگوهای مقاوم‌سازی (Resilience Patterns) در معماری Microservices، استفاده همزمان از Retry و Circuit Breaker است.

این دو الگو مکمل یکدیگر هستند. Retry تلاش می‌کند خطاهای موقتی را برطرف کند، در حالی که Circuit Breaker از ارسال درخواست‌های بی‌فایده به سرویس‌های از کار افتاده جلوگیری می‌کند.

💡
Retry برای خطاهای موقتی مناسب است، اما اگر سرویس مقصد واقعاً Down باشد، Retry به تنهایی فقط باعث افزایش بار سیستم خواهد شد.

نمونه پیاده‌سازی با Polly

Retry + Circuit Breaker 
builder.Services
    .AddHttpClient("Payment")
    .AddTransientHttpErrorPolicy(policy =>
        policy.WaitAndRetryAsync(
            3,
            retryAttempt =>
                TimeSpan.FromSeconds(
                    Math.Pow(2, retryAttempt))))
    .AddTransientHttpErrorPolicy(policy =>
        policy.CircuitBreakerAsync(
            handledEventsAllowedBeforeBreaking: 5,
            durationOfBreak: TimeSpan.FromSeconds(30)));

توضیح پارامترها

پارامتر توضیح
handledEventsAllowedBeforeBreaking تعداد خطاهای مجاز قبل از باز شدن Circuit
durationOfBreak مدت زمانی که Circuit باز می‌ماند.

در مثال بالا:

  • اگر ۵ درخواست متوالی با خطا مواجه شوند...
  • Circuit به مدت ۳۰ ثانیه باز خواهد شد.
  • در این مدت هیچ درخواستی به Payment ارسال نمی‌شود.
  • پس از ۳۰ ثانیه چند درخواست آزمایشی ارسال خواهد شد.

Best Practices

  • Retry را فقط برای خطاهای موقتی انجام دهید.
  • همیشه از Exponential Backoff استفاده کنید.
  • Retry را با Circuit Breaker ترکیب کنید.

🎯
Retry و Circuit Breaker از مهم‌ترین الگوهای Resilience در معماری Microservices هستند. Retry خطاهای موقتی را مدیریت می‌کند و Circuit Breaker از گسترش خرابی (Cascading Failure) جلوگیری می‌کند. در پروژه‌های ASP.NET Core، معمولاً این دو الگو همراه با IHttpClientFactory و Polly یا راهکارهای جدید Resilience Framework استفاده می‌شوند.

Best Practices

  • در ASP.NET Core همیشه از IHttpClientFactory استفاده کنید.
  • برای هر سرویس خارجی یک Typed Client یا Named Client در نظر بگیرید.
  • از BaseAddress و DefaultRequestHeaders در زمان ثبت Client استفاده کنید.
  • از Polly برای Retry، Timeout و Circuit Breaker بهره ببرید.
  • Timeoutها را متناسب با نیاز هر سرویس تنظیم کنید.
  • در Retry از Exponential Backoff استفاده کنید.
  • درخواست‌ها را با CancellationToken قابل لغو طراحی کنید.
  • منطق فراخوانی API را در Controllerها قرار ندهید و آن را در Clientهای اختصاصی یا سرویس‌های لایه Infrastructure متمرکز کنید.
🎯
در پروژه‌های Enterprise مبتنی بر ASP.NET Core، ترکیب IHttpClientFactory + Typed Client + Polly به‌عنوان یکی از بهترین الگوهای پیاده‌سازی ارتباط‌های REST شناخته می‌شود. این ترکیب علاوه بر افزایش خوانایی کد، قابلیت اطمینان، کارایی و نگهداری سیستم را بهبود می‌بخشد.

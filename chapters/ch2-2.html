2.2 gRPC

👤 تیم فنی

پس از آشنایی با ارتباط REST، نوبت به یکی از سریع‌ترین روش‌های ارتباط بین Microserviceها می‌رسد.

gRPC یک فریمورک متن‌باز برای ایجاد ارتباط Remote Procedure Call (RPC) است که توسط Google توسعه داده شده و امروزه یکی از محبوب‌ترین روش‌های ارتباط بین سرویس‌های داخلی (Internal Services) محسوب می‌شود.

در پروژه‌های Enterprise، به‌ویژه در معماری Microservices، زمانی که سرعت، کارایی و مصرف کم پهنای باند اهمیت زیادی داشته باشد، gRPC معمولاً جایگزین مناسبی برای REST خواهد بود.

💡
REST برای ارتباط با Browser و Mobile بسیار مناسب است، اما برای ارتباط Service به Service معمولاً gRPC انتخاب بهتری محسوب می‌شود.

gRPC چیست؟

gRPC مخفف عبارت Google Remote Procedure Call است.

در gRPC به جای اینکه مستقیماً با URLها و Endpointها کار کنیم، متدهای یک سرویس را مانند متدهای یک کلاس فراخوانی می‌کنیم.

Concept 

Order Service

      │

      ▼

در ظاهر، فراخوانی یک متد انجام می‌شود، اما در پشت صحنه اطلاعات از طریق شبکه منتقل می‌شوند.

✅
gRPC پیچیدگی ارتباط شبکه را پنهان می‌کند و ارتباط بین سرویس‌ها را شبیه فراخوانی یک متد محلی (Local Method Call) نشان می‌دهد.

چرا gRPC؟

REST تقریباً در تمام زبان‌ها و Platformها پشتیبانی می‌شود، اما استفاده از JSON و HTTP/1.1 باعث ایجاد سربار (Overhead) می‌شود.

در پروژه‌هایی که هزاران درخواست بین Microserviceها رد و بدل می‌شود، این سربار می‌تواند Performance سیستم را کاهش دهد.

gRPC برای حل همین مشکل طراحی شده است.

به همین دلیل بسیاری از شرکت‌های بزرگ مانند Google، Netflix، Microsoft، Dropbox و VMware از gRPC برای ارتباط داخلی سرویس‌های خود استفاده می‌کنند.

HTTP/2

یکی از مهم‌ترین دلایل سرعت بالای gRPC، استفاده از HTTP/2 است.

برخلاف REST که معمولاً روی HTTP/1.1 اجرا می‌شود، gRPC به صورت پیش‌فرض از HTTP/2 استفاده می‌کند.

📌
بخش بزرگی از افزایش Performance در gRPC به دلیل قابلیت‌های HTTP/2 است، نه فقط استفاده از Protocol Buffers.

Multiplexing

در HTTP/1.1 معمولاً هر Connection فقط یک درخواست را همزمان پردازش می‌کند.

در HTTP/2 چندین درخواست می‌توانند همزمان روی یک Connection ارسال شوند.

Binary Protocol

REST معمولاً داده‌ها را به صورت JSON ارسال می‌کند که یک فرمت متنی (Text) است.

اما gRPC از Protocol Buffers (Protobuf) استفاده می‌کند که یک فرمت Binary است.

⚠️
اگر API شما توسط مرورگر یا سرویس‌های خارجی مصرف می‌شود، REST معمولاً انتخاب بهتری است. اما برای ارتباط داخلی بین Microserviceها، gRPC اغلب کارایی و مصرف منابع بهتری ارائه می‌دهد.

مزایای gRPC

  • سرعت بسیار بالا
  • استفاده از HTTP/2
  • حجم بسیار کم پیام‌ها
  • Serialization سریع با Protocol Buffers
  • پشتیبانی بومی از Streaming
  • Code Generation خودکار
  • Type Safety بالا
  • مناسب برای ارتباط Service to Service
  • مصرف کمتر CPU و پهنای باند نسبت به JSON
✅
در معماری‌های مبتنی بر Microservices، gRPC یکی از بهترین گزینه‌ها برای ارتباط‌های همزمان (Synchronous Communication) با حجم بالای درخواست است.

معایب gRPC

  • وابستگی به HTTP/2
  • خواندن داده‌های Binary دشوارتر از JSON است.
  • تست دستی با ابزارهایی مانند مرورگر ساده نیست.
  • نیاز به فایل Proto برای تعریف قرارداد سرویس
  • یادگیری اولیه نسبت به REST پیچیده‌تر است.
  • پشتیبانی محدودتر در برخی Gatewayها و مرورگرها
⚠️
gRPC جایگزین REST نیست؛ بلکه مکمل آن است. در بسیاری از سیستم‌های Enterprise، REST برای APIهای عمومی و gRPC برای ارتباط داخلی بین Microserviceها به صورت همزمان استفاده می‌شوند.
🎯
در بخش بعد با Protocol Buffers (Proto File) آشنا خواهیم شد و خواهیم دید چگونه تنها با نوشتن یک فایل .proto، کلاس‌های C#، Serviceها و Clientهای gRPC به صورت خودکار تولید می‌شوند.

Protocol Buffers (Proto File)

قلب gRPC فایل Proto است.

برخلاف REST که قرارداد ارتباطی معمولاً به صورت مستندات Swagger یا OpenAPI تعریف می‌شود، در gRPC تمام قرارداد ارتباطی داخل فایل .proto قرار می‌گیرد.

این فایل مشخص می‌کند:

  • چه Serviceهایی وجود دارند.
  • هر Service چه متدهایی دارد.
  • ورودی هر متد چیست.
  • خروجی هر متد چیست.
  • چه Enumها و Messageهایی وجود دارند.
💡
فایل Proto قرارداد (Contract) بین Client و Server است. تا زمانی که این فایل یکسان باشد، هر دو طرف بدون توجه به زبان برنامه‌نویسی می‌توانند با یکدیگر ارتباط برقرار کنند.

ساختار یک فایل Proto

payment.proto 
syntax = "proto3";

package payment;

option csharp_namespace = "Payment.Grpc";

message PaymentRequest
{
    int32 id = 1;
}

message PaymentResponse
{
    int32 id = 1;
    string status = 2;
    double amount = 3;
}

service PaymentService
{
    rpc GetPayment(PaymentRequest)
        returns (PaymentResponse);
}

اولین خط هر فایل Proto نسخه سینتکس را مشخص می‌کند.

✅

Service

Service مجموعه‌ای از متدهای قابل فراخوانی را مشخص می‌کند.

    rpc CreatePayment(CreatePaymentRequest)
        returns (CreatePaymentResponse);

هر Service تقریباً معادل یک Controller در REST است.

Best Practices

  • همیشه از proto3 استفاده کنید.
  • شماره Fieldها را بعد از انتشار API تغییر ندهید.
  • برای حذف یک Property، شماره آن را دوباره استفاده نکنید.
  • از Messageهای کوچک و مستقل استفاده کنید.
  • برای وضعیت‌ها از Enum استفاده کنید.
  • از Package و csharp_namespace مناسب استفاده کنید.
  • Proto را به عنوان Contract مشترک بین تیم‌ها نگهداری کنید.
  • فایل‌های Proto را Version Control کنید.
  • از نام‌گذاری واضح برای Service و Messageها استفاده کنید (مانند PaymentService و PaymentRequest).
  • در صورت نیاز به توسعه API، Fieldهای جدید را با شماره‌های جدید اضافه کنید و از تغییر یا حذف شماره‌های موجود خودداری کنید تا سازگاری با نسخه‌های قبلی حفظ شود.
🎯
در بخش بعد با چهار مدل اصلی ارتباط در gRPC یعنی Unary، Server Streaming، Client Streaming و Bidirectional Streaming آشنا خواهیم شد و خواهیم دید هر کدام در چه سناریوهایی استفاده می‌شوند.

انواع ارتباط در gRPC

برخلاف REST که تنها بر پایه الگوی Request / Response کار می‌کند، gRPC چهار مدل مختلف ارتباط را در اختیار برنامه‌نویس قرار می‌دهد.

این موضوع یکی از مهم‌ترین مزیت‌های gRPC نسبت به REST است، زیرا برای هر سناریو می‌توان مناسب‌ترین مدل ارتباطی را انتخاب کرد.

نوع ارتباط توضیح REST معادل
Unary RPC یک درخواست، یک پاسخ ✔ مشابه REST
Server Streaming یک درخواست، چند پاسخ ✖
Client Streaming چند درخواست، یک پاسخ ✖
Bidirectional Streaming چند درخواست، چند پاسخ ✖
💡
بیشتر پروژه‌های Enterprise حدود ۸۰ تا ۹۰ درصد از ارتباطات خود را با Unary RPC پیاده‌سازی می‌کنند و تنها در سناریوهای خاص از Streaming استفاده می‌شود.

مزایای Unary RPC

  • ساده‌ترین مدل ارتباط در gRPC
  • پیاده‌سازی آسان
  • شباهت زیاد به REST
  • Performance بالاتر نسبت به REST
  • استفاده از HTTP/2 و Protocol Buffers
  • مناسب برای اغلب عملیات CRUD
  • Code Generation خودکار
  • Type Safety بالا
  • Latency کمتر نسبت به REST
🚀
اگر تنها به یک Request و یک Response نیاز دارید، Unary RPC تقریباً همیشه اولین انتخاب در gRPC است.

معایب Unary RPC

  • عدم پشتیبانی از ارسال چند پاسخ
  • برای داده‌های لحظه‌ای مناسب نیست.
  • برای Live Streaming کاربرد ندارد.
  • برای انتقال حجم زیاد داده به صورت تدریجی مناسب نیست.
  • برای ارتباط‌های بلادرنگ (Real-Time) گزینه ایده‌آلی نیست.
⚠️
اگر لازم است Server چندین پیام را به مرور زمان برای Client ارسال کند، به جای Unary باید از Server Streaming استفاده کنید که در بخش بعد بررسی خواهیم کرد.

مثال واقعی (OrderService → PaymentService)

فرض کنید در یک سیستم فروشگاهی، کاربر سفارشی را ثبت می‌کند.

پس از ثبت سفارش، سرویس سفارش (Order Service) باید وضعیت پرداخت را از Payment Service استعلام بگیرد.

چون تنها به یک درخواست و یک پاسخ نیاز داریم، بهترین گزینه استفاده از Unary RPC است.

💡
اگر Payment Service فقط یک نتیجه برگرداند (مانند وضعیت پرداخت)، Unary RPC ساده‌ترین و سریع‌ترین انتخاب است.

چه زمانی Unary مناسب نیست؟

اگر Server مجبور باشد چندین پیام را در طول زمان برای Client ارسال کند، Unary دیگر انتخاب مناسبی نیست.

جمع‌بندی

Unary RPC رایج‌ترین و پرکاربردترین نوع ارتباط در gRPC است و از نظر مفهومی شباهت زیادی به مدل Request/Response در REST دارد؛ با این تفاوت که از HTTP/2 و Protocol Buffers استفاده می‌کند و معمولاً سرعت و کارایی بالاتری ارائه می‌دهد.

در بیشتر پروژه‌های مبتنی بر Microservices، عملیات‌هایی مانند دریافت اطلاعات، استعلام وضعیت، احراز هویت، ثبت سفارش و سایر درخواست‌هایی که تنها به یک پاسخ نیاز دارند، با Unary RPC پیاده‌سازی می‌شوند.

🎯
می‌توان Unary RPC را معادل gRPC برای الگوی Request/Response در REST در نظر گرفت؛ با این تفاوت که ارتباط سریع‌تر، کم‌حجم‌تر و Type-Safe است.

Best Practices

  • برای عملیات‌های Request/Response از Unary RPC استفاده کنید.
  • از Unary برای عملیات CRUD بین Microserviceها استفاده کنید.
  • برای پیام‌های کوچک و پاسخ‌های سریع، Unary بهترین انتخاب است.
  • در صورت نیاز به ارسال مداوم داده، به جای Unary از Streaming استفاده کنید.
  • برای هر RPC، Deadline و CancellationToken تعریف کنید تا از انتظار بی‌پایان جلوگیری شود.
  • در عملیات‌هایی که ممکن است Retry شوند، منطق سرویس را Idempotent طراحی کنید.
  • خطاهای gRPC را با Status Codeهای مناسب (مانند NotFound، InvalidArgument و Unavailable) مدیریت کنید.
  • برای ارتباط Service-to-Service در شبکه داخلی، Unary RPC معمولاً انتخاب پیش‌فرض و توصیه‌شده است.
🚀
در بخش بعد با Server Streaming RPC آشنا خواهیم شد؛ مدلی که در آن Client تنها یک درخواست ارسال می‌کند، اما Server می‌تواند چندین پاسخ را به‌صورت پیوسته و Real-Time برای Client ارسال کند. این مدل برای سناریوهایی مانند Live Price، Monitoring، Notification و Log Streaming بسیار مناسب است.

Proto File

در gRPC تمام قرارداد ارتباطی (Contract) بین Client و Server داخل فایل .proto تعریف می‌شود.

این فایل مشخص می‌کند:

  • چه Serviceهایی وجود دارند.
  • هر Service چه متدهایی دارد.
  • ورودی هر متد چیست.
  • خروجی هر متد چیست.
  • کلاس‌های C# چگونه تولید شوند.

فرض کنید می‌خواهیم Order Service وضعیت پرداخت را از Payment Service دریافت کند.

payment.proto 
syntax = "proto3";

option csharp_namespace = "Payment.Grpc";

package payment;

service PaymentService
{
    rpc GetPaymentStatus (PaymentRequest)
        returns (PaymentResponse);
}

message PaymentRequest
{
    int32 orderId = 1;
}

message PaymentResponse
{
    int32 orderId = 1;

    bool success = 2;

    string transactionId = 3;

    double amount = 4;

    PaymentStatus status = 5;
}

enum PaymentStatus
{
    Pending = 0;

    Success = 1;

    Failed = 2;

    Cancelled = 3;
}
💡
فایل Proto تنها فایل مشترک بین Client و Server است و هر دو پروژه از روی همین فایل کلاس‌های موردنیاز خود را تولید می‌کنند.

پیاده‌سازی Service در ASP.NET Core

پس از Build پروژه، یک کلاس پایه به نام PaymentService.PaymentServiceBase تولید می‌شود.

برای پیاده‌سازی Service کافی است از این کلاس ارث‌بری کنیم.

PaymentGrpcService.cs 
using Grpc.Core;

namespace Payment.Api.Services;

public class PaymentGrpcService
    : PaymentService.PaymentServiceBase
{
    public override Task<PaymentResponse>
        GetPaymentStatus(
            PaymentRequest request,
            ServerCallContext context)
    {
        var response = new PaymentResponse
        {
            OrderId = request.OrderId,
            Success = true,
            TransactionId = Guid.NewGuid().ToString(),
            Amount = 250,
            Status = PaymentStatus.Success
        };

        return Task.FromResult(response);
    }
}

در این مثال:

  • Client شناسه سفارش را ارسال می‌کند.
  • Service اطلاعات پرداخت را تولید می‌کند.
  • یک شیء PaymentResponse به Client برگردانده می‌شود.
✅
در عمل، به‌جای داده‌های نمونه، این اطلاعات معمولاً از Database یا یک سرویس دامنه (Domain Service) خوانده می‌شوند.

ثبت Service در Program.cs

برای اینکه gRPC درخواست‌ها را به Service موردنظر هدایت کند، باید آن را در Program.cs ثبت کنیم.

Program.cs 
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddGrpc();

var app = builder.Build();

app.MapGrpcService<PaymentGrpcService>();

app.MapGet("/",
    () => "Use a gRPC client.");

app.Run();
کد توضیح
AddGrpc() فعال‌سازی سرویس‌های gRPC در DI Container
MapGrpcService() ثبت Endpoint مربوط به Service
MapGet() نمایش پیام برای درخواست‌های HTTP معمولی
Run() اجرای Web Application
🎯
تا این مرحله، Service سمت سرور آماده است. در بخش بعد، نحوه ایجاد gRPC Client در Order Service، ثبت آن در Dependency Injection و فراخوانی متد GetPaymentStatus() را بررسی خواهیم کرد.

    PaymentService.PaymentServiceClient>(options =>
{
    options.Address =
        new Uri("https://localhost:7002");

در اینجا:

Best Practices

  • Client را همیشه با AddGrpcClient() ثبت کنید و از ایجاد دستی Channel در هر درخواست خودداری کنید.
  • آدرس سرویس را در فایل‌های پیکربندی (appsettings.json) نگهداری کنید، نه به‌صورت Hard Code.
  • از Dependency Injection برای دریافت Client استفاده کنید.
  • برای تمام فراخوانی‌ها CancellationToken و Deadline تعریف کنید تا درخواست‌ها بی‌نهایت منتظر نمانند.
  • خطاهای gRPC را با RpcException مدیریت کنید و بر اساس StatusCode تصمیم‌گیری کنید.
  • در صورت نیاز به احراز هویت، Metadata یا Bearer Token را همراه درخواست ارسال کنید.
  • منطق فراخوانی gRPC را در لایه Infrastructure یا Clientهای اختصاصی قرار دهید و Controllerها را ساده نگه دارید.
  • برای ارتباط‌های داخلی بین Microserviceها از TLS و HTTP/2 استفاده کنید.
  • برای عملیات‌هایی که ممکن است Retry شوند، منطق سرویس را Idempotent طراحی کنید.
🎯
اکنون با پیاده‌سازی کامل یک ارتباط Unary در gRPC آشنا شدید. در بخش بعد، Server Streaming را بررسی خواهیم کرد؛ مدلی که در آن Client تنها یک درخواست ارسال می‌کند، اما Server می‌تواند مجموعه‌ای از پاسخ‌ها را به‌صورت پیوسته و Real-Time برای Client ارسال کند.

Server Streaming چیست؟

در بخش قبل با Unary RPC آشنا شدیم که در آن Client یک درخواست ارسال می‌کرد و Server نیز تنها یک پاسخ برمی‌گرداند.

اما همیشه این مدل پاسخگوی نیاز سیستم نیست.

گاهی Server باید پس از دریافت یک درخواست، چندین پیام را در طول زمان برای Client ارسال کند.

در چنین شرایطی از Server Streaming RPC استفاده می‌کنیم.

💡
Server Streaming یعنی: One Request → Multiple Responses یعنی Client فقط یک درخواست ارسال می‌کند اما Server می‌تواند هر زمان که لازم باشد چندین Response برای همان درخواست ارسال کند.

پیاده‌سازی Server Streaming در ASP.NET Core

پس از تعریف فایل .proto و Build شدن پروژه، کلاس پایه Service به صورت خودکار تولید می‌شود.

برای پیاده‌سازی Server Streaming کافی است از کلاس تولید شده ارث‌بری کنیم و به جای برگرداندن یک شیء، پاسخ‌ها را از طریق IServerStreamWriter<T> برای Client ارسال کنیم.

💡
در Unary مقدار Response را Return می‌کردیم، اما در Server Streaming پاسخ‌ها یکی یکی توسط Server نوشته (Write) می‌شوند.

ساخت Service

PriceGrpcService.cs 
using Grpc.Core;

namespace Price.Api.Services;

public class PriceGrpcService
    : PriceService.PriceServiceBase
{
    public override async Task StreamPrices(
        PriceRequest request,
        IServerStreamWriter<PriceResponse> responseStream,
        ServerCallContext context)
    {

    }
}

همانند Unary، کلاس از PriceServiceBase ارث‌بری می‌کند.

اما این بار به جای خروجی Task<PriceResponse> تنها یک Task برگردانده می‌شود.

دلیل آن این است که Responseها توسط responseStream ارسال خواهند شد.

ارسال چندین پیام

در مثال زیر هر ثانیه یک قیمت جدید برای Client ارسال می‌شود.

Streaming Service 
using Grpc.Core;

namespace Price.Api.Services;

public class PriceGrpcService
    : PriceService.PriceServiceBase
{
    public override async Task StreamPrices(
        PriceRequest request,
        IServerStreamWriter<PriceResponse> responseStream,
        ServerCallContext context)
    {
        var random = new Random();

        for (int i = 0; i < 10; i++)
        {
            var response = new PriceResponse
            {
                Symbol = request.Symbol,
                Price = 50000 + random.NextDouble() * 1000,
                Time = DateTime.Now.ToString("HH:mm:ss")
            };

            await responseStream.WriteAsync(response);

            await Task.Delay(1000);
        }
    }
}

در این مثال:

  • Client فقط یک درخواست ارسال می‌کند.
  • Server ده پیام مختلف ارسال می‌کند.
  • بین هر پیام یک ثانیه فاصله وجود دارد.
  • پس از پایان حلقه، ارتباط بسته می‌شود.
✅
هر بار اجرای WriteAsync() معادل ارسال یک Response جدید برای همان Client است.

پیاده‌سازی Server Streaming در ASP.NET Core

پس از تعریف فایل .proto و Build شدن پروژه، کلاس پایه Service به صورت خودکار تولید می‌شود.

برای پیاده‌سازی Server Streaming کافی است از کلاس تولید شده ارث‌بری کنیم و به جای برگرداندن یک شیء، پاسخ‌ها را از طریق IServerStreamWriter<T> برای Client ارسال کنیم.

💡
در Unary مقدار Response را Return می‌کردیم، اما در Server Streaming پاسخ‌ها یکی یکی توسط Server نوشته (Write) می‌شوند.

Streaming Service 
using Grpc.Core;

namespace Price.Api.Services;

public class PriceGrpcService
    : PriceService.PriceServiceBase
{
    public override async Task StreamPrices(
        PriceRequest request,
        IServerStreamWriter<PriceResponse> responseStream,
        ServerCallContext context)
    {
        var random = new Random();

        for (int i = 0; i < 10; i++)
        {
            var response = new PriceResponse
            {
                Symbol = request.Symbol,
                Price = 50000 + random.NextDouble() * 1000,
                Time = DateTime.Now.ToString("HH:mm:ss")
            };

            await responseStream.WriteAsync(response);

            await Task.Delay(1000);
        }
    }
}

  • Client فقط یک درخواست ارسال می‌کند.
  • Server ده پیام مختلف ارسال می‌کند.
  • بین هر پیام یک ثانیه فاصله وجود دارد.
  • پس از پایان حلقه، ارتباط بسته می‌شود.
✅
هر بار اجرای WriteAsync() معادل ارسال یک Response جدید برای همان Client است.

خطا هنگام ارسال پیام

در صورتی که هنگام ارسال پیام خطایی رخ دهد یا Client ارتباط را قطع کند، ممکن است متد WriteAsync() استثناء ایجاد کند.

Error Handling 
try
{
    await responseStream.WriteAsync(response);
}
catch (RpcException)
{
    // Client disconnected
}

در پروژه‌های واقعی بهتر است این خطاها ثبت (Log) شوند تا بتوان وضعیت ارتباط Clientها را بررسی کرد.

Best Practices

  • برای ارسال هر پیام از WriteAsync() استفاده کنید.
  • هرگز از return response در Server Streaming استفاده نکنید.
  • همیشه context.CancellationToken را بررسی کنید.
  • از await برای تمام فراخوانی‌های WriteAsync() استفاده کنید.
  • در صورت ارسال داده‌های حجیم، بین پیام‌ها فاصله منطقی ایجاد کنید تا از فشار بیش از حد بر Client و شبکه جلوگیری شود.
  • منابعی مانند Connectionهای پایگاه داده یا فایل‌ها را پس از پایان Stream آزاد کنید.
  • خطاهای ناشی از قطع ارتباط Client را مدیریت و ثبت (Logging) کنید.
  • برای Streamهای طولانی، از Deadline و Timeout مناسب استفاده کنید.
  • در صورت نیاز به ارسال حجم بسیار زیاد داده، Paging یا Batch کردن پیام‌ها را در نظر بگیرید.
🚀
IServerStreamWriter<T> مهم‌ترین جزء Server Streaming است. این Interface امکان ارسال تدریجی و Real-Time داده‌ها را فراهم می‌کند و به Server اجازه می‌دهد تا زمانی که ارتباط برقرار است، هر تعداد پیام را بدون ایجاد درخواست جدید برای Client ارسال کند.

پیاده‌سازی Client در ASP.NET Core

در Server Streaming، سمت Client برخلاف Unary تنها یک Response دریافت نمی‌کند.

پس از ارسال Request، یک Stream از پیام‌ها دریافت می‌شود که باید به صورت تدریجی خوانده شوند.

💡
در Unary از await client.GetSomethingAsync() استفاده می‌کردیم، اما در Server Streaming باید پیام‌ها را یکی یکی از Stream بخوانیم.

خواندن Stream

پس از ارسال درخواست، متد Streaming یک AsyncServerStreamingCall<T> برمی‌گرداند.

Client 
public class PriceClient
{
    private readonly PriceService
        .PriceServiceClient _client;

    public PriceClient(
        PriceService.PriceServiceClient client)
    {
        _client = client;
    }

    public async Task StreamPrices()
    {
        using var call =
            _client.StreamPrices(
                new PriceRequest
                {
                    Symbol = "BTCUSDT"
                });

        await foreach (var item in
            call.ResponseStream.ReadAllAsync())
        {
            Console.WriteLine(
                $"{item.Symbol} : {item.Price}");
        }
    }
}

به محض اینکه Server یک پیام جدید ارسال کند، حلقه await foreach آن را دریافت خواهد کرد.

✅
Client منتظر پایان Stream نمی‌ماند؛ هر پیام بلافاصله پس از ارسال توسط Server قابل پردازش است.

Best Practices

  • از await foreach برای خواندن Stream استفاده کنید.
  • همیشه شیء AsyncServerStreamingCall را با using یا await using مدیریت کنید تا منابع آزاد شوند.
  • در صورت امکان از CancellationToken برای توقف دریافت Stream استفاده کنید.
  • منطق پردازش هر پیام را سبک نگه دارید تا دریافت پیام‌های بعدی به تأخیر نیفتد.
  • در Streamهای طولانی، خطاهای شبکه و RpcException را مدیریت کنید.
  • برای داده‌های لحظه‌ای مانند قیمت سهام، لاگ‌ها، اعلان‌ها و Monitoring از Server Streaming استفاده کنید.
  • اگر فقط به یک پاسخ نیاز دارید، از Unary RPC استفاده کنید تا پیچیدگی سیستم کاهش یابد.
  • در ارتباط‌های طولانی‌مدت، Timeout و Deadline مناسب تعریف کنید تا در صورت قطع ارتباط یا بروز خطا، منابع سیستم آزاد شوند.
🚀
Server Streaming یکی از قدرتمندترین قابلیت‌های gRPC است و امکان ارسال داده‌های Real-Time را تنها با یک Request فراهم می‌کند. این الگو برای سناریوهایی مانند Live Price، Notification، Monitoring، Log Streaming و Telemetry بسیار مناسب است و جایگزینی کارآمد برای Polling مداوم یا درخواست‌های تکراری REST به شمار می‌رود.
🎯
در بخش بعد با Client Streaming RPC آشنا می‌شویم؛ مدلی که در آن Client می‌تواند چندین پیام را برای Server ارسال کند و در پایان تنها یک پاسخ دریافت نماید. این الگو برای سناریوهایی مانند آپلود فایل، ارسال Batch داده‌ها و Import اطلاعات بسیار کاربردی است.

Ack در RabbitMQ

📅 به‌روزرسانی: 1405/04/30
⏱️ زمان مطالعه: ۹ دقیقه
👤 تیم فنی

وقتی RabbitMQ یک پیام را برای Consumer ارسال می‌کند، باید مطمئن شود پیام با موفقیت پردازش شده است. این تأیید با مفهومی به نام Acknowledgment یا به‌اختصار Ack انجام می‌شود.

در سناریوی ما، Notification Service پیام OrderCreated را دریافت کرده و پس از ارسال موفق پیامک تأیید سفارش، آن را Ack می‌کند.

Message Acknowledgment Flow 

Order Service
      │
      │ OrderCreated
      ▼
RabbitMQ Queue
      │
      ▼
Notification Service
      │
      ├── پردازش موفق ──► Ack
      │
      └── پردازش ناموفق ──► Nack / Reject
🎯
RabbitMQ با دریافت Ack متوجه می‌شود که پیام با موفقیت پردازش شده و می‌تواند آن را از Queue حذف کند.

Automatic Ack

اگر autoAck برابر true باشد، RabbitMQ بلافاصله بعد از ارسال پیام به Consumer، آن را پردازش‌شده در نظر می‌گیرد.

C# - Automatic Ack 
var consumer = new AsyncEventingBasicConsumer(channel);

consumer.ReceivedAsync += async (_, eventArgs) =>
{
    string json = Encoding.UTF8.GetString(
        eventArgs.Body.Span);

    await SendOrderNotificationAsync(json);
};

await channel.BasicConsumeAsync(
    queue: "order.notification.queue",
    autoAck: true,
    consumer: consumer);
⚠️
اگر Consumer پس از دریافت پیام و قبل از پایان پردازش متوقف شود، پیام از دست می‌رود؛ زیرا RabbitMQ قبلاً آن را تأییدشده در نظر گرفته است.

Manual Ack

در حالت Manual Ack مقدار autoAck برابر false قرار می‌گیرد و Consumer فقط پس از پردازش موفق، پیام را تأیید می‌کند.

C# - Manual Ack 
using RabbitMQ.Client.Events;
using System.Text;

const string queueName = "order.notification.queue";

var consumer = new AsyncEventingBasicConsumer(channel);

consumer.ReceivedAsync += async (_, eventArgs) =>
{
    string json = Encoding.UTF8.GetString(
        eventArgs.Body.Span);

    await SendOrderNotificationAsync(json);

    await channel.BasicAckAsync(
        deliveryTag: eventArgs.DeliveryTag,
        multiple: false);
};

await channel.BasicConsumeAsync(
    queue: queueName,
    autoAck: false,
    consumer: consumer);

مقدار DeliveryTag شناسه پیام در همان Channel است و برای Ack یا Nack کردن پیام استفاده می‌شود.

✅
برای پیام‌های مهم مانند OrderCreated معمولاً استفاده از Manual Ack انتخاب مطمئن‌تری است.

BasicAckAsync

متد BasicAckAsync به RabbitMQ اعلام می‌کند که پیام با موفقیت پردازش شده است.

C# - Basic Ack 
await channel.BasicAckAsync(
    deliveryTag: eventArgs.DeliveryTag,
    multiple: false);
پارامتر توضیح
deliveryTag شناسه پیام تحویل‌داده‌شده در Channel
multiple مشخص می‌کند یک پیام یا چند پیام قبلی نیز تأیید شوند.

وقتی multiple برابر false باشد، فقط همان پیام تأیید می‌شود.

BasicNackAsync

اگر پردازش پیام ناموفق باشد، می‌توان با BasicNackAsync پیام را رد کرد.

C# - Basic Nack 
await channel.BasicNackAsync(
    deliveryTag: eventArgs.DeliveryTag,
    multiple: false,
    requeue: false);

پارامتر requeue تعیین می‌کند پیام دوباره به Queue برگردد یا خیر.

مقدار requeue رفتار
true پیام دوباره وارد Queue می‌شود.
false پیام حذف یا به Dead Letter Exchange منتقل می‌شود.
⚠️
استفاده بدون محدودیت از requeue: true ممکن است باعث ایجاد حلقه بی‌نهایت شود و همان پیام بارها پردازش شود.

BasicRejectAsync

متد BasicRejectAsync نیز برای رد کردن پیام استفاده می‌شود، اما فقط یک پیام را رد می‌کند و پارامتر multiple ندارد.

C# - Basic Reject 
await channel.BasicRejectAsync(
    deliveryTag: eventArgs.DeliveryTag,
    requeue: false);
متد کاربرد
BasicAckAsync تأیید پردازش موفق پیام
BasicNackAsync رد کردن یک یا چند پیام
BasicRejectAsync رد کردن یک پیام

مدیریت خطا در Consumer

بهتر است Ack فقط بعد از پایان کامل عملیات ارسال شود. در صورت رخ دادن خطا نیز پیام Nack می‌شود.

C# - OrderCreated Consumer 
var consumer = new AsyncEventingBasicConsumer(channel);

consumer.ReceivedAsync += async (_, eventArgs) =>
{
    try
    {
        string json = Encoding.UTF8.GetString(
            eventArgs.Body.Span);

        var orderCreated =
            JsonSerializer.Deserialize<OrderCreatedMessage>(json);

        if (orderCreated is null)
            throw new InvalidOperationException(
                "OrderCreated message is invalid.");

        await SendOrderNotificationAsync(orderCreated);

        await channel.BasicAckAsync(
            deliveryTag: eventArgs.DeliveryTag,
            multiple: false);
    }
    catch (Exception exception)
    {
        Console.WriteLine(
            $"OrderCreated processing failed: {exception.Message}");

        await channel.BasicNackAsync(
            deliveryTag: eventArgs.DeliveryTag,
            multiple: false,
            requeue: false);
    }
};

await channel.BasicConsumeAsync(
    queue: "order.notification.queue",
    autoAck: false,
    consumer: consumer);
📌
قرار دادن Ack داخل finally اشتباه است؛ زیرا حتی در صورت شکست پردازش نیز پیام تأیید و از Queue حذف می‌شود.

اگر Consumer قطع شود چه اتفاقی می‌افتد؟

اگر Consumer قبل از ارسال Ack قطع شود، RabbitMQ پیام تأییدنشده را دوباره در اختیار یک Consumer فعال قرار می‌دهد.

⚠️
تحویل مجدد پیام ممکن است باعث اجرای دوباره عملیات شود. به همین دلیل Consumer باید Idempotent طراحی شود.

بررسی Redelivered

ویژگی Redelivered نشان می‌دهد RabbitMQ احتمالاً پیام را قبلاً نیز به یک Consumer تحویل داده است.

C# - Redelivered Flag 
consumer.ReceivedAsync += async (_, eventArgs) =>
{
    if (eventArgs.Redelivered)
    {
        Console.WriteLine(
            "This message has been delivered before.");
    }

    // Process OrderCreated message...
};
💡
مقدار Redelivered به‌تنهایی جایگزین Idempotency نیست. برای جلوگیری از پردازش تکراری بهتر است MessageId در دیتابیس یا Cache ذخیره و بررسی شود.

Ack همراه با Prefetch

Prefetch مشخص می‌کند هر Consumer حداکثر چند پیام تأییدنشده دریافت کند.

C# - Prefetch Count 
await channel.BasicQosAsync(
    prefetchSize: 0,
    prefetchCount: 10,
    global: false);

در این مثال، Notification Service حداکثر ۱۰ پیام بدون Ack را هم‌زمان در اختیار خواهد داشت.

✅
ترکیب Manual Ack و Prefetch مناسب باعث کنترل بار Consumer و توزیع بهتر پیام‌ها می‌شود.

اشتباهات رایج

  • Ack کردن پیام قبل از پایان پردازش
  • استفاده از autoAck برای پیام‌های مهم
  • بازگرداندن همیشگی پیام خطادار با requeue: true
  • نادیده گرفتن احتمال تحویل تکراری پیام

نکته مصاحبه

🎓
سؤال: اگر Consumer قبل از ارسال Ack متوقف شود چه اتفاقی می‌افتد؟

پاسخ: RabbitMQ پیام را تأییدنشده در نظر می‌گیرد و پس از بسته شدن Connection یا Channel، آن را دوباره برای یک Consumer فعال ارسال می‌کند.

قانون طلایی

🎯
پیام را فقط زمانی Ack کنید که تمام عملیات موردنیاز با موفقیت انجام شده باشد. همچنین Consumer را برای پردازش تکراری پیام آماده کنید.

جمع‌بندی

Ack به RabbitMQ اعلام می‌کند که پیام با موفقیت پردازش شده است. برای پیام‌های مهم بهتر است autoAck غیرفعال باشد و پس از پایان پردازش از BasicAckAsync استفاده شود.

در صورت شکست پردازش می‌توان پیام را با BasicNackAsync یا BasicRejectAsync رد کرد. تصمیم درباره Requeue نیز باید همراه با Retry محدود و Dead Letter Queue انجام شود.

➡️
در بخش بعد Retry را بررسی می‌کنیم و یاد می‌گیریم پیام‌های ناموفق OrderCreated چگونه بدون ایجاد حلقه بی‌نهایت دوباره پردازش شوند.

Producer در Kafka

📅 به‌روزرسانی: 1405/04/31
⏱️ زمان مطالعه: ۹ دقیقه
👤 تیم فنی

Producer سرویسی است که Message یا Event را تولید کرده و در یک Topic از Kafka منتشر می‌کند.

Producer مشخص می‌کند پیام در کدام Topic قرار بگیرد و می‌تواند با استفاده از Message Key روی انتخاب Partition نیز تأثیر بگذارد.

🎯
وظیفه Producer فقط ارسال Event نیست؛ نحوه انتخاب Key، سطح Acknowledgment و مدیریت Retry مستقیماً بر Performance و قابلیت اطمینان سیستم اثر می‌گذارد.

جریان ارسال Message

Producer Flow 

Application

Producer ابتدا Event را Serialize می‌کند، سپس Topic و Partition مقصد را مشخص کرده و Message را برای Broker ارسال می‌کند.

ساختار Message

هر Message در Kafka می‌تواند شامل Key، Value، Header و Timestamp باشد.

Kafka Message 

Message

   ├── Key
   ├── Value
   ├── Headers
   └── Timestamp
بخش کاربرد
Key انتخاب Partition و حفظ ترتیب Eventهای مرتبط
Value محتوای اصلی Event
Headers اطلاعات جانبی مانند CorrelationId و Version
Timestamp زمان ایجاد یا ثبت Message

ارسال Event در .NET

برای کار با Kafka در .NET معمولاً از پکیج Confluent.Kafka استفاده می‌شود.

.NET CLI 
dotnet add package Confluent.Kafka
C# Producer 
using Confluent.Kafka;
using System.Text.Json;

var producerConfig = new ProducerConfig
{
    BootstrapServers = "localhost:9092",
    Acks = Acks.All
};

using var producer =
    new ProducerBuilder<string, string>(producerConfig).Build();

var orderCreated = new
{
    EventId = Guid.NewGuid(),
    OrderId = 1001,
    CustomerId = 25,
    TotalAmount = 850000,
    OccurredAt = DateTime.UtcNow
};

var message = new Message<string, string>
{
    Key = orderCreated.OrderId.ToString(),
    Value = JsonSerializer.Serialize(orderCreated)
};

var deliveryResult = await producer.ProduceAsync(
    "order-created",
    message
);

Console.WriteLine(
    $"Topic: {deliveryResult.Topic}, " +
    $"Partition: {deliveryResult.Partition}, " +
    $"Offset: {deliveryResult.Offset}"
);

متد ProduceAsync پس از ارسال Message، اطلاعات Topic، Partition و Offset ثبت‌شده را برمی‌گرداند.

✅
بهتر است Producer به‌صورت Singleton ساخته شود و برای هر Message نمونه جدیدی از آن ایجاد نشود.

Producer به‌صورت سرویس در ASP.NET Core

KafkaProducer.cs 
using Confluent.Kafka;
using System.Text.Json;

public sealed class KafkaProducer : IDisposable
{
    private readonly IProducer<string, string> _producer;

    public KafkaProducer(IConfiguration configuration)
    {
        var producerConfig = new ProducerConfig
        {
            BootstrapServers =
                configuration["Kafka:BootstrapServers"],

            Acks = Acks.All,
            EnableIdempotence = true
        };

        _producer = new ProducerBuilder<string, string>(
            producerConfig
        ).Build();
    }

    public async Task PublishAsync<T>(
        string topic,
        string key,
        T value,
        CancellationToken cancellationToken = default)
    {
        var message = new Message<string, string>
        {
            Key = key,
            Value = JsonSerializer.Serialize(value)
        };

        await _producer.ProduceAsync(
            topic,
            message,
            cancellationToken
        );
    }

    public void Dispose()
    {
        _producer.Flush(TimeSpan.FromSeconds(10));
        _producer.Dispose();
    }
}
Program.cs 
builder.Services.AddSingleton<KafkaProducer>();
appsettings.json 
{
  "Kafka": {
    "BootstrapServers": "localhost:9092"
  }
}

استفاده در Order Service

OrderService.cs 
public sealed class OrderService
{
    private readonly KafkaProducer _producer;

    public OrderService(KafkaProducer producer)
    {
        _producer = producer;
    }

    public async Task CreateOrderAsync(
        int orderId,
        decimal totalAmount,
        CancellationToken cancellationToken)
    {
        // Save order in database

        var orderCreated = new
        {
            EventId = Guid.NewGuid(),
            OrderId = orderId,
            TotalAmount = totalAmount,
            OccurredAt = DateTime.UtcNow
        };

        await _producer.PublishAsync(
            topic: "order-created",
            key: orderId.ToString(),
            value: orderCreated,
            cancellationToken
        );
    }
}
⚠️
ذخیره Order در Database و انتشار Event دو عملیات جدا هستند. برای جلوگیری از ذخیره موفق Order و شکست انتشار Event، معمولاً از الگوی Transactional Outbox استفاده می‌شود.

Message Key

Message Key مشخص می‌کند Event معمولاً در کدام Partition قرار بگیرد.

Message Key 

OrderId: 1001

OrderCreated
OrderPaid
OrderShipped

      │

Same Key: 1001

      ▼

Same Partition

قرار گرفتن Eventهای مربوط به یک Order در یک Partition، امکان حفظ ترتیب آن‌ها را فراهم می‌کند.

⚠️
استفاده از یک Key ثابت برای تمام Messageها باعث می‌شود همه Eventها وارد یک Partition شوند و پردازش موازی کاهش پیدا کند.

Acknowledgment

تنظیم Acks مشخص می‌کند Producer برای موفق تلقی کردن ارسال Message منتظر چه سطحی از تأیید Broker بماند.

مقدار رفتار
Acks.None Producer منتظر تأیید Broker نمی‌ماند.
Acks.Leader پس از ذخیره Message توسط Leader تأیید دریافت می‌شود.
Acks.All Leader منتظر تأیید Replicaهای همگام می‌ماند.
📌
Acks.All قابلیت اطمینان بیشتری دارد، اما ممکن است Latency بیشتری نسبت به تنظیمات ضعیف‌تر ایجاد کند.

Idempotent Producer

ممکن است Producer پس از Timeout یا خطای شبکه همان Message را دوباره ارسال کند و باعث ایجاد Duplicate شود.

با فعال کردن EnableIdempotence Kafka می‌تواند از ثبت تکراری برخی ارسال‌های مجدد Producer جلوگیری کند.

Idempotent Producer 
var config = new ProducerConfig
{
    BootstrapServers = "localhost:9092",
    Acks = Acks.All,
    EnableIdempotence = true
};
⚠️
Idempotent Producer فقط ارسال به Kafka را پوشش می‌دهد. Consumer همچنان باید برای جلوگیری از اجرای دوباره عملیات Business به‌صورت Idempotent طراحی شود.

Headers

برای ارسال اطلاعات جانبی مانند CorrelationId، نوع Event و نسخه قرارداد می‌توان از Header استفاده کرد.

Message Headers 
using System.Text;

var message = new Message<string, string>
{
    Key = orderId.ToString(),
    Value = JsonSerializer.Serialize(orderCreated),
    Headers = new Headers
    {
        {
            "correlation-id",
            Encoding.UTF8.GetBytes(Guid.NewGuid().ToString())
        },
        {
            "event-version",
            Encoding.UTF8.GetBytes("1")
        }
    }
};
💡
CorrelationId برای Distributed Tracing و پیدا کردن جریان یک درخواست میان چند Microservice بسیار مفید است.

مدیریت خطای ارسال

Error Handling 
try
{
    var result = await producer.ProduceAsync(
        "order-created",
        message,
        cancellationToken
    );

    Console.WriteLine(
        $"Published at {result.TopicPartitionOffset}"
    );
}
catch (ProduceException<string, string> exception)
{
    Console.WriteLine(
        $"Kafka publish failed: {exception.Error.Reason}"
    );

    throw;
}

خطای انتشار نباید نادیده گرفته شود. این خطا باید Log شده و براساس سناریوی سیستم، Retry یا ذخیره برای پردازش مجدد انجام شود.

بهبود Performance با Batching

Kafka Producer می‌تواند چند Message را جمع‌آوری کرده و آن‌ها را به‌صورت Batch ارسال کند.

تنظیم کاربرد
LingerMs مدت کوتاهی برای جمع‌آوری Messageهای بیشتر منتظر می‌ماند.
BatchSize حداکثر اندازه تقریبی Batch را مشخص می‌کند.
CompressionType حجم داده ارسالی را با Compression کاهش می‌دهد.
Producer Performance 
var config = new ProducerConfig
{
    BootstrapServers = "localhost:9092",
    Acks = Acks.All,
    EnableIdempotence = true,
    LingerMs = 5,
    BatchSize = 32 * 1024,
    CompressionType = CompressionType.Snappy
};
📌
افزایش Batching معمولاً Throughput را بیشتر می‌کند، اما ممکن است Latency ارسال هر Message را کمی افزایش دهد.

Best Practiceها

انجام دهید ✅ انجام ندهید ❌
Producer را به‌صورت Singleton استفاده کنید. برای هر Message یک Producer جدید نسازید.
Message Key مناسب انتخاب کنید. برای تمام Messageها Key ثابت قرار ندهید.
خطای ProduceAsync را مدیریت کنید. ارسال موفق Message را بدون بررسی فرض نکنید.
برای داده‌های مهم از Acks.All استفاده کنید. فقط برای Performance قابلیت اطمینان را حذف نکنید.
از Outbox Pattern استفاده کنید. Database و Kafka را یک Transaction واحد فرض نکنید.

اشتباهات رایج

❌
ایجاد Producer جدید برای هر درخواست باعث مصرف Connection و کاهش Performance می‌شود.
❌
استفاده نکردن از Message Key مناسب می‌تواند Ordering موردنیاز Business را از بین ببرد.
❌
انتشار Event قبل از ذخیره موفق اطلاعات در Database ممکن است Eventی ایجاد کند که وضعیت واقعی سیستم را نمایش نمی‌دهد.

سؤالات مصاحبه

🎓
سؤال ۱: Producer در Kafka چه وظیفه‌ای دارد؟

پاسخ: Producer مسئول Serialize کردن و ارسال Message به Topic مناسب است و می‌تواند با استفاده از Key روی انتخاب Partition تأثیر بگذارد.
🎓
سؤال ۲: Message Key چه کاربردی دارد؟

پاسخ: Message Key برای انتخاب Partition استفاده می‌شود. Messageهایی با Key یکسان معمولاً در یک Partition قرار می‌گیرند و ترتیب آن‌ها حفظ می‌شود.
🎓
سؤال ۳: تفاوت Acks.Leader و Acks.All چیست؟

پاسخ: در Acks.Leader پس از ثبت Message توسط Leader پاسخ موفق دریافت می‌شود؛ اما در Acks.All، Leader منتظر تأیید Replicaهای همگام نیز می‌ماند.
🎓
سؤال ۴: EnableIdempotence چه کاربردی دارد؟

پاسخ: از ثبت تکراری Messageهایی که به دلیل Retry مجدداً توسط همان Producer ارسال شده‌اند جلوگیری می‌کند.
🎓
سؤال ۵: چرا Kafka Producer بهتر است Singleton باشد؟

پاسخ: Producer برای مدیریت Connection، Batching و ارسال Message طراحی شده و ساخت مداوم آن باعث مصرف منابع و کاهش Performance می‌شود.
🎓
سؤال ۶: چگونه مشکل ذخیره موفق داده و شکست انتشار Event را حل می‌کنید؟

پاسخ: با استفاده از Transactional Outbox، داده Business و Event در یک Transaction داخل Database ذخیره می‌شوند و Worker جداگانه Event را به Kafka منتشر می‌کند.

قانون طلایی

🎯
Producer باید طوری طراحی شود که ارسال Message قابل اعتماد باشد، اما موفقیت انتشار Event را با موفقیت کامل عملیات Business اشتباه نگیرد.

جمع‌بندی

Kafka Producer مسئول تولید و انتشار Event در Topic است. Message Key، تنظیم Acks، Retry، Batching و Idempotence از مهم‌ترین تنظیمات Producer هستند.

در پروژه‌های .NET بهتر است Producer به‌صورت Singleton ثبت شود، خطاهای ارسال مدیریت شوند و برای هماهنگی Database و Kafka از Transactional Outbox استفاده شود.
